# ember v0.6 integration notes — the first real embedding-target integration

The v1.0 example game-engine integration (`examples/game_host.cpp` +
`examples/scripts/game_logic.ember`) is the first end-to-end proof that ember
works as a game-logic embedding target. It exercises every host integration seam:
`@entry` setup, `@on_tick` per-frame over host state, `@event("player_input")`
response, native bindings operating on host-owned entity state, and hot reload
of a gameplay fn mid-loop. 12/12 ctest green.

## What it proved (the embeddability seams)

- **Lifecycle** (v0.6 `lifecycle.hpp`): `get_entry_function` + `get_annotated_functions("@on_tick")` + `get_event_handlers("player_input")` discover the script's entry/tick/event fns; the host calls them via the dispatch table. Works in a real host, not just a test harness.
- **Native bindings** (v0.3 `BindingBuilder`): the host registers `get_entity_count`/`get/set_entity_x`/`get/set_speed`/`get_delta_time`/`log` operating on a host-side `std::vector<Entity>` + a host float. The script reads/writes host state through these — the i64-handle/slice convention the six extensions already use. **No fluent TypeBuilder/StructBuilder/engine_t needed.**
- **Hot reload** (v0.6 `hot_reload.hpp`): mid-loop, the host reloads the `update` fn to move entities LEFT instead of right; the dispatch slot swaps and the next 5 frames use the new body. Per-function independence holds in a real host context.
- **`--tick` lifecycle mode** (v0.6 `ember_cli --tick`): a separate proof that the tick lifecycle works on the terminal runtime, shared shape with this host.

## The v1.0 fluent-API decision (the integration was the trigger test)

The v0.3 deferred-binding analysis (`docs/v0.3_DEFERRED_BINDING_ANALYSIS.md`)
concluded the fluent `TypeBuilder`/`StructBuilder`/`engine_t` surface is
trigger-gated on "a host wants script-visible C++ struct types." This
integration is the first real host. **It used BindingBuilder + i64 handles +
StructLayoutTable for all natives and hit NO binding friction.** The trigger
remains unfired. The fluent API stays deferred to v1.0+ — now with evidence
from the first real integration, not just the extension audit.

If a future host wants to expose a C++ struct with fields the script reads/
writes directly (not through get/set natives), THAT fires the trigger — and the
use case will shape the TypeBuilder API. Until then, the i64-handle + native-
access pattern is sufficient (it's what AngelScript's scriptdictionary and the
surveyed native-JIT language's addons do too).

## A real codegen bug the integration found

The integration exposed a codegen bug: **float global WRITES are broken.** A
script global of type `f32`, written via `g = <float expr>`, does not land in
the globals block in the real (heap-allocated GlobalsBlock) host case — the
store goes to the wrong place / is lost. The bug:

- `codegen.cpp`'s AssignExpr global-store path called `store_rax_to_global`
  (stores RAX, a 64-bit GP store) even for float globals — but a float value is
  in **xmm0**, not rax. So `g = 10.0f` stored garbage (whatever was in rax).
- **Partial fix applied:** added `store_xmm0_to_global` (movss [r11+disp32],
  xmm0, with the correct REX.B for r11) and routed float-global stores to it.
  This makes float-global writes work in the stack-GlobalsBlock case (verified
  by an isolated probe: `g = 10.0f` → gbs[0] = 10.0f).

- **Remaining gap:** in the heap-allocated GlobalsBlock case (the real host
  shape — `unique_ptr<GameModule>` with `GlobalsBlock gb` inside the heap
  struct), the float-global store still segfaults. The stack case works; the
  heap case doesn't. This points to a stale `g_globals_for_codegen` pointer or
  a base-address resolution issue specific to the heap layout, and is **not
  fully resolved**. The integration works around it by holding the speed in
  the HOST (get_speed/set_speed natives) rather than a script f32 global —
  which is also the more idiomatic game-scripting pattern (host owns the
  authoritative state; the script reads/writes it through natives).

- **Also found:** ember does not evaluate global *initializers* at load (the
  globals block is zero-initialized; `global g : f32 = 10.0f` starts at 0, not
  10). A host or `@entry` must seed globals. This is a known gap, not a
  regression — documented here for the first time because the integration is
  the first real consumer of globals.

Both are tracked for a follow-up codegen pass. The integration is otherwise
complete and proves embeddability.
