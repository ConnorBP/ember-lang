# Ember v1.0 Audit Report — 2026-07-09

**Audited revision:** `381ed76` (`Docs: v1.0 concurrency + Tier 2 batch update pass`)  
**Primary platform:** Windows x86-64, MinGW g++ 15.2.0, Ninja, C++17, Release (`-O3 -DNDEBUG`)  
**Scope:** frontend, sema, JIT/codegen, Win64 ABI, `.em` serialization/loading, traps and safety controls, modules/hot reload, extensions, concurrency, embedding APIs, tests, benchmark methodology, and documentation.

## Executive summary

A clean Release build completed **17/17 CTest targets in 21.58 seconds** at the start of the audit. That baseline is useful, but it does not establish language or sandbox correctness: several shipped language paths are parse/sema-only in the suite, malformed `.em` input is almost untested, and the SPSC stress test contains a producer-loop bug that can lose values or hang nondeterministically.

The audit confirmed strong foundations: executable memory follows RW→RX W^X discipline; the B1 per-thread `context_t` model isolates independent callers; function-reference range/allowlist checks are real; `PERM_FFI` is enforced by sema; dispatch publication is atomic; and the module registry's base-storage invariant is stable after construction.

It also found release-blocking defects. The most serious are:

1. malformed `.em` relocation offsets can bypass a 32-bit bounds check and perform an out-of-bounds write;
2. attacker-controlled `.em` sizes/counts trigger multi-gigabyte allocations, hangs, and uncaught allocation exceptions;
3. unrestricted casts can forge slices and turn scalar values into host pointers;
4. integer-compatible named handles can be forged and then null-dereferenced by extension operators;
5. extension size arguments can become enormous `size_t` allocations and let C++ exceptions escape the Ember trap model;
6. generated `switch` code and B1 entry wrappers violate Win64 nonvolatile-register rules;
7. multiple documented primitive and control-flow features either emit incorrect code or emit no code.

No production source was changed during this audit. Temporary reproductions are under the ignored `tmp_edit/` directory.

### Finding summary

| Tier | Count | Meaning |
|---|---:|---|
| Critical | 5 | Reproduced memory-safety or host-termination defect across a claimed input/sandbox boundary |
| High | 15 | Reproduced wrong-code/ABI defect or inspection-confirmed unsafe lifetime/serialization issue |
| Medium | 13 | Contract-gated concurrency issue, incomplete safety mechanism, API/test/build defect, or narrower semantic mismatch |
| Documentation/test gaps | 11 groups | Stale or contradictory claims and missing evidence |
| Static-analysis candidates | 3 | Plausible UB/invariant issues not dynamically reproduced in this audit |

Severity does **not** imply that a `.em` file is a safe untrusted-code container. `.em` stores native x86-64 and a successfully loaded module can already execute native instructions. The loader findings are nevertheless critical robustness defects because its public contract promises malformed input returns `false` rather than corrupting or terminating the host.

### Recommended remediation order

1. Harden `.em` parsing before every allocation or pointer calculation; add malformed-input/fuzz tests.
2. Enforce cast and opaque-handle type barriers; validate every extension handle and size before dereference/allocation.
3. Preserve Win64 nonvolatile registers and add an ABI-preservation test for `rbx`, `rsi`, `rdi`, `r12`–`r15`, and XMM6–XMM15.
4. Fix or explicitly remove unimplemented shipped language paths (`do`, `sizeof`, `offsetof`, division traps, widths/signedness).
5. Fix the SPSC stress test, then restore a deterministic 17/17 release gate.
6. Make `.em` portability/trust guarantees explicit and relocate or symbolically bind every process-local address.
7. Reconcile stable docs with the implementation and label unimplemented design sections as deferred.

## 1. Methodology and classification

Four read-only audit tracks covered implementation quality, security, performance, and documentation/tests. Findings were then cross-checked against source and, for high-risk paths, reduced to small executable probes.

Evidence labels used below:

- **Reproduced:** executed against the audited Release build or a minimal probe linked to it.
- **Confirmed by inspection:** direct control/data-flow evidence exists, but the trigger needs a special host, concurrent schedule, or ABI-sensitive stub.
- **Contract-gated:** unsafe only when a documented trusted-host or single-threaded contract is violated.
- **Documented deferral:** intentionally not implemented and not classified as a defect unless another document claims it shipped.

The audit did **not** include exhaustive fuzzing, ASan/UBSan instrumentation, a current MSVC build, formal verification of emitted instructions, or performance measurements across multiple machines.

## 2. Critical findings

### C1. `.em` relocation-offset wrap causes an out-of-bounds write

**Evidence:** Reproduced; `tmp_edit/em_reloc_wrap.em` exits with signal/return code 139.

`EmReloc::offset` is `uint32_t`. `src/em_loader.cpp` validates it with:

```cpp
if (r.offset + 8 > f.code.size()) { ... }
```

For an offset such as `0xfffffffc`, the addition wraps in 32 bits. The check passes, after which `page_bytes + r.offset` is used as the destination of an eight-byte relocation write.

Reproduction:

```bash
cd ember
./tmp_edit/audit_em_loader_probe.exe tmp_edit/em_reloc_wrap.em
echo $?
# Segmentation fault; rc=139
```

**Impact:** malformed input writes outside the allocated RW code page before W^X sealing. Because `.em` is native code, this is principally a loader memory-safety and contract violation, not a claim that `.em` is an untrusted-code sandbox.

**Fix:** avoid additive bounds checks in the narrow type. Require `code.size() >= 8 && uint64_t(r.offset) <= uint64_t(code.size()) - 8`. Validate all relocation records before allocating or writing any executable page.

### C2. `.em` counts and sizes drive unbounded allocations and uncaught exceptions

**Evidence:** Reproduced.

Disk-controlled values reach allocation APIs without format limits or exception containment:

- `out.globals.resize(global_size)`;
- `fns.reserve(function_count)`;
- `f.relocs.resize(reloc_count)`;
- `out.dispatch.assign(max_slot + 1, nullptr)`;
- `out.pages.reserve(function_count)`;
- `out.name_table.reserve(name_table_count)`.

Observed probes:

```bash
./tmp_edit/audit_em_loader_probe.exe tmp_edit/em_huge_fc.em
# THREW std::exception: std::bad_alloc; rc=100

./tmp_edit/audit_em_loader_probe.exe tmp_edit/em_huge_globals.em
# err=em_loader: truncated read ... globals=4294967295

# em_huge_slot.em and em_huge_reloc.em attempt enormous allocation/work
# and exceeded the audit timeout.
```

This directly contradicts `src/em_loader.hpp`, which states that malformed files never throw and all failures return `false` with an error.

**Impact:** process termination, severe memory pressure, or denial of service from malformed modules.

**Fix:** define format-level maxima for file size, function count, slots, globals, code/rodata per function, relocation count, and name-table count. Check multiplication/addition overflow and remaining file length before allocation. Catch `std::bad_alloc`, `std::length_error`, and parse exceptions at the loader boundary and return a structured error.

### C3. Arbitrary casts can forge slices and host pointers

**Evidence:** Reproduced by the frontend/codegen audit; accepted by `sema_check`, then segfaulted with rc=139 when indexed.

`src/sema.cpp::Checker::check_expr(CastExpr&)` accepts cast pairs without applying the conversion matrix. `src/codegen.cpp::CG::eval(CastExpr)` implements integer/float conversions, but other casts are effectively representation-preserving. A scalar can therefore become the pointer word of a two-word slice while the length word is uncontrolled.

```ember
fn main() -> i64 {
    let x: i64 = 1;
    let s: u8[] = x as u8[];
    return s[0];
}
```

**Impact:** script code can construct invalid raw pointers despite the documented no-raw-pointer/type-safety model, causing arbitrary memory reads/writes or host termination.

**Fix:** implement an explicit cast-compatibility table in sema. Reject all scalar↔slice, scalar↔aggregate, unrelated-handle, and representation reinterpretations. Codegen should assert that every remaining cast is one of its implemented conversions.

### C4. Opaque extension handles are forgeable and operator paths dereference null

**Evidence:** Reproduced; `tmp_edit/forge_vec.ember` passes sema and exits 139.

```ember
fn main() -> i64 {
    let bad: vec3 = 999;
    let good = vec3_new(1.0f, 2.0f, 3.0f);
    let z = bad + good;
    return 0;
}
```

The broad “any integer is compatible with any integer-backed type” rule allows an `i64` literal to initialize the named `vec3` handle type. Operator overloads in `extensions/vec/ext_vec.cpp` dereference the result of handle lookup without checking for `nullptr`. Equivalent unchecked operator paths exist in `extensions/quat/ext_quat.cpp` and `extensions/mat/ext_mat.cpp`.

**Impact:** ordinary script input can terminate the host through a null dereference. Stale handles after reset and forged handles arriving through a host/native boundary have the same effect.

**Fix:** make named opaque handles semantically distinct from integers; only their constructor/native result should produce the type. Require explicit, privileged conversion if a host truly needs one. Independently null-check every extension lookup and return/trap consistently. Both layers are required for defense in depth.

### C5. Negative/overflowing extension sizes escape as C++ allocation exceptions

**Evidence:** Reproduced; `tmp_edit/negative_array.ember` prints an uncaught `std::length_error` and exits abnormally.

```ember
fn main() -> i64 {
    let h = array_new(1, -1);
    return h;
}
```

`extensions/array/ext_array.cpp` converts expressions such as `elem_size * count` and `n * elem_size` to `size_t` without first rejecting negative values or checking multiplication overflow. Allocation exceptions escape the JIT/native call and are outside Ember's recoverable trap surface.

**Impact:** script-controlled process termination and potentially extreme allocation attempts.

**Fix:** reject negative sizes; use checked `size_t` multiplication; impose per-container and host-wide limits; catch allocation errors at the extension boundary and convert them to a documented Ember error/trap. Audit string, sync, and other allocation-taking extensions for the same pattern.

## 3. High findings

### H1. `do-while` parses and sema-checks but emits no code

**Evidence:** Reproduced.

`parser.cpp` builds `DoWhileStmt` and sema visits it, but `CG::exec_stmt` has no `DoWhileStmt` branch. The entire body and condition disappear:

```ember
fn main() -> i64 {
    do { return 7; } while (false);
    return 3;
}
```

Observed result: `3`, not `7`. A mutating two-iteration example returned `0` instead of `2`.

**Fix:** implement bottom-tested lowering with `continue` targeting the condition and a budget check on the taken back edge, or reject the syntax until implemented. Add execution tests, not only parse/sema tests.

### H2. Integer division/modulo bypass the unified trap model

**Evidence:** Reproduced; runtime division by zero terminated abnormally instead of returning a `TrapReason`.

Integer `/`, `%`, and compound forms emit bare `cqo; idiv rcx` in `src/codegen.cpp`. There is no guard for divisor zero or `INT64_MIN / -1`, despite `CODEGEN_SPEC.md` and `SAFETY_AND_SANDBOX.md` requiring an always-on recoverable trap.

**Fix:** emit explicit zero/overflow branches before `idiv`, route them through the context trap stub, and add both normal and B1 tests for `/`, `%`, `/=`, and `%=`.

### H3. Generated `switch` code clobbers Win64 nonvolatile `r13`

**Evidence:** Reproduced with `tmp_edit/test_r13.cpp`.

`switch` lowering stores its subject in `r13`; the JIT prologue/epilogue preserves `rbx` but not `r13`.

```bash
./tmp_edit/test_r13.exe
# result=11 before=1122334455667788 after=1
# rc=1
```

**Impact:** host or caller state can be silently corrupted after a correct-looking script return.

**Fix:** avoid nonvolatile scratch registers, or save/restore each one in a fixed frame slot. Add a host ABI test that seeds and checks all Win64 nonvolatile GP and XMM registers around representative generated functions.

### H4. B1 entry wrappers clobber the caller's nonvolatile `r14`

**Evidence:** Confirmed by inspection and disassembly; a dedicated audit probe also observed `r14` changing to the `context_t*`.

`src/engine.cpp::ember_call_void` and `ember_call_i64` write `r14` in inline assembly but neither save/restore its incoming value nor declare it as a clobber. As exported C++ functions they must preserve `r14` for their caller.

The wrappers also rely on GNU inline assembly and therefore do not implement the advertised MSVC x64 path.

**Fix:** provide a small ABI-correct assembly thunk per supported compiler, preserving `r14` and reserving correct Win64 shadow/alignment space, or save/restore `r14` explicitly with compiler-visible constraints. Test generated disassembly and register preservation.

### H5. Trap-stub calls are misaligned at most trap sites

**Evidence:** Confirmed by inspection; latent with current simple stubs.

At ordinary JIT body sites `rsp` is 16-byte aligned before a Win64 call. `emit_trap()` subtracts `0x28`, leaving the call site misaligned; only paths with a preceding push happen to have the opposite parity. The shipped stubs tolerate this, but an optimized stub using aligned SSE stack spills may fault instead of trapping.

**Fix:** make stack parity an explicit emitter invariant. Standardize trap sites at one known alignment and reserve 32 bytes of shadow space plus only the padding actually required. Add an alignment-sensitive trap stub test.

### H6. `sizeof` returns an undefined runtime value; `offsetof` is not parsed

**Evidence:** Reproduced. `sizeof(i64)` sema-checks as `u64`, but no codegen branch produces its value; one probe returned 176 rather than 8. `Kw_offsetof` exists in the lexer but has no parser/AST implementation.

**Fix:** constant-fold both operators before codegen using the finalized layout table, or reject them. Add primitive, struct, nested-struct, fixed-array, and invalid-field cases.

### H7. `f64` executes as `f32`

**Evidence:** Reproduced. All float literal/arithmetic/comparison paths use `movss`, `ucomiss`, and single-precision conversion. `16777217.0 > 16777216.0` evaluated false.

`TYPE_SYSTEM.md` explicitly specifies IEEE-754 binary64, so this is not a documented v1 simplification.

**Fix:** either implement distinct scalar width/type-directed SSE paths (`movsd`, `ucomisd`, `cvtsi2sd`, etc.) or remove `f64` from the accepted v1 language and docs until supported.

### H8. Unsigned and narrow integer semantics are not implemented

**Evidence:** Reproduced.

- unsigned comparisons use signed conditions (`UINT64_MAX > 0` evaluated false);
- unsigned division uses `idiv`;
- right shift always uses arithmetic `sar` (`u64(1<<63) >> 63` yielded `-1`);
- narrow arithmetic is not truncated (`u8(255)+1` did not wrap to zero).

**Fix:** make codegen type-directed for signed comparison/division/right-shift and normalize/truncate results at declared widths. Add a table-driven boundary suite for every integer type.

### H9. Sema permits narrowing and signedness-changing implicit conversions

**Evidence:** Reproduced. `let y: i8 = x_i64;` is accepted because all integer types are considered compatible, without inserting a conversion node.

This contradicts `TYPE_SYSTEM.md`, which allows only lossless same-signedness widening implicitly.

**Fix:** centralize assignment/call/return conversion rules and insert explicit typed conversion nodes for allowed widening. Require `as` for narrowing or signedness changes.

### H10. Recorded-signature function-reference calls skip argument-type validation

**Evidence:** Reproduced. A handle to `fn(i64,i64)->i64` accepted two `f32` values. Sema supplies expected-type hints but never compares the checked argument types to the recorded signature, so codegen places values in XMM registers while the callee reads GP registers.

This is separate from the documented bare-`fn` signature hole.

**Fix:** enforce each recorded parameter type after expression checking and apply only legal conversions. Add wrong-type tests for integer, float, slice, struct, and fn-handle parameters.

### H11. Local-array slice escape analysis is absent

**Evidence:** Confirmed by accepted reproduction.

```ember
fn bad() -> u8[] {
    let a: u8[1];
    return a[..];
}
```

The view is typed as a slice, but return/global assignment does not track provenance. The result points into a dead JIT stack frame.

**Fix:** implement the escape rejection claimed by `COMPILER_PIPELINE.md`, or disallow returning/storing all views of local fixed arrays until provenance exists.

### H12. `.em` output embeds process-local pointers and can lose initialized globals

**Evidence:** Confirmed by inspection.

Raw immediates are emitted for native function targets, trap stub/detail pointers, function-reference allowlist storage, and some string storage. `.em` relocation kinds cover only dispatch, globals, and module-registry bases. Loading such output in another process/build can call or dereference stale addresses.

In addition, the CLI `--emit-em` path builds a fresh zero-filled globals buffer instead of serializing the initialized `gb_store`, so constant initializers can revert to zero.

**Impact:** real modules are not reliably reusable across processes even though the format is presented as precompiled native output. A malicious `.em` requires no pointer trick to execute native code; the issue here is portability, integrity, and loud rejection of incompatible artifacts.

**Fix:** serialize symbolic native/import identities and resolve them through a host-provided allowlisted binding table; put strings in serialized rodata; relocate trap/allowlist state or prohibit those features in `.em`; add ABI/build IDs and optional authentication; serialize the actual initialized global bytes. Test emit in process A and load in process B.

### H13. Hot reload accepts incompatible signatures

**Evidence:** Reproduced by the runtime audit: `fn f() -> i64` was accepted as a reload of `fn f(x:i64) -> i64`.

`reload_function()` matches the function name and reuses its slot but does not enforce the comment/spec requirement that parameters and return type remain identical.

**Impact:** existing call sites and cached host callers invoke the replacement using the old ABI.

**Fix:** compare canonical parameter/return types and calling-shape metadata before compile/publication; add changed-arity, changed-parameter, changed-return, and annotation tests.

### H14. `.em` linking has no export-signature verification

**Evidence:** Confirmed by inspection.

`src/module_linker.hpp::build_em_exports()` marks every export `unknown_sig`, causing sema to skip arity/return checking. The current `.em` format does not store signatures, so `MODULES.md`'s promised link-time verification cannot occur.

**Fix:** version the format to include canonical export/import signatures and reject mismatches before JIT execution. Until then, document `.em` imports as ABI-trusted and remove the stronger guarantee.

### H15. The SPSC stress test is itself incorrect and can fail or hang

**Evidence:** Reproduced by the runtime audit; direct execution reported FIFO loss (`expected 3888 got 4407`), while one CTest run left `ext_sync` blocked.

`examples/ext_sync_test.cpp` loops on the function's return value:

```cpp
while (!spsc_push_host(h, i, &pushed)) {}
```

But the function returns whether the handle is valid; queue fullness is returned through `out_pushed`. A full queue therefore drops the value and advances the producer loop. A dropped tail value can leave the consumer waiting forever. A second multithreaded case can print worker failures without propagating them to the test result.

**Impact:** the advertised 17-test release gate is nondeterministic and does not prove no-loss queue behavior, despite one clean baseline pass.

**Fix:** loop until `pushed == true`, propagate all worker failures through atomics/futures, add bounded timeouts, and run the stress repeatedly in CI.

## 4. Medium findings

### M1. `ext_sync` store vectors race across independent contexts

The global vectors/free lists in `extensions/sync/ext_sync.cpp` are mutated by allocation/free/reset and read by accessors without synchronization. Per-context single-threading does not protect two contexts from a `push_back` reallocation racing another context's slot lookup. Protect cold store-management operations with a mutex or use stable, atomically published slots. Primitive operations on an already acquired stable slot can remain fast.

### M2. Lifecycle's “approximate snapshot” is not C++-race-safe

`extensions/lifecycle/ext_lifecycle.cpp` iterates `g_routines` while registration may `push_back`. Approximate semantics do not make concurrent vector mutation legal; reallocation can produce use-after-free. Synchronize registration/snapshot or publish immutable snapshots. The CLI's current sequencing avoids the race, so this is contract-gated there.

### M3. Multi-context hot reload has no in-flight reclamation

Atomic slot replacement is correct, but `reload_function()` returns the old executable page and provides no epoch/quiescence implementation. Freeing it while another context executes it is a use-after-free; retaining forever leaks. This is honestly described as a single-threaded CLI limitation in source, but broader docs present epoch reclamation as if shipped. Keep this a documented deferral or implement host-visible epochs.

### M4. Budget/depth accounting is weaker than the safety specification

Budget checks occur at `while`/`for` back edges but not function entry, so long straight-line and call-heavy execution can avoid instruction accounting. Native calls are omitted from depth accounting. Either align the docs to “loop budget + script-call depth” or implement the stated entry/native costs.

### M5. `defer` is function-exit scoped, not lexical scope-exit scoped

Codegen tracks defer actions at function level. It does not execute them on nested-block fallthrough, `break`, or `continue`, and a defer inside a loop runs at most once. This contradicts the documented per-block LIFO semantics. Implement cleanup edges or narrow the feature/docs.

### M6. Postfix increment/decrement return the new value

Parser desugaring makes prefix and postfix forms equivalent. A probe `x=1; y=x++; return y*10+x;` returned 22 rather than 12. Preserve the old value for postfix forms and add side-effect-order tests.

### M7. `switch` fallthrough semantics disagree across code, tests, and docs

Sema permits fallthrough and codegen emits adjacent case bodies. Stable docs state that implicit fallthrough is rejected, while `sema_valid_switch_full.ember` expects it. Choose one language rule and make parser/sema/codegen/tests/docs consistent.

### M8. Top-level `const` is discarded

The parser recognizes `const` and then stores a `GlobalDecl` with no constness field. Assignments to the global remain legal. Preserve mutability in the AST/symbol table and reject stores, or remove top-level `const` from the grammar.

### M9. Several invalid forms pass sema

Same-scope redeclarations, assignment to non-lvalues, `break`/`continue` outside loops, recursive by-value structs, and `void` locals were accepted in audit probes. Strengthen scope declarations, lvalue classification, control-context tracking, and type validation; add one negative test per invariant.

### M10. Struct layout and >8-byte argument ABI do not match the specification

Struct fields are tightly concatenated without required alignment/trailing padding; nested layout can fall back to size 8; recursive structs lack cycle rejection. Large by-value arguments are expanded into multiple Ember words rather than passed according to the Win64 aggregate rule. Define whether script structs are host-C-layout or Ember-layout, then make bindings and documentation match. Expand `binding_abi_test` beyond an 8-byte argument and large return.

### M11. Aggregate globals receive only one eight-byte slot

The global table and load/store paths assume one word, but slices need pointer+length and structs/arrays need their full layout. Sema accepts aggregate global types. Reject unsupported aggregate globals or allocate typed offsets/sizes and implement complete initialization/load/store.

### M12. `LoadedModule` is copyable despite owning raw executable pages

A user-declared destructor frees all `pages`, but copy construction/assignment are not deleted and implicit move generation is suppressed. Copying duplicates ownership and can double-free executable pages. Make the type explicitly move-only and store pages in a move-only RAII wrapper.

### M13. Malformed hex can abort compiler tooling

`0x` with no digits reaches `std::stoull` with an empty string; `invalid_argument` is not converted to a lexer diagnostic. Decimal suffixes are consumed but not reflected in type semantics, and hex suffix handling is incomplete. Replace exception-driven parsing with checked digit accumulation or catch all conversion failures; test suffixes and malformed boundaries.

## 5. API, build, and CLI discrepancies

These are not all runtime defects, but they materially affect integration quality.

1. **`ember_call_void` / `ember_call_i64` do not establish a recoverable checkpoint.** They only install `r14` and call. Every example manually duplicates reset/setjmp/error handling, while `context.hpp` describes `ember_call` as establishing the checkpoint. Provide one reusable safe-call API or rename/document the current helpers as raw B1 thunks.
2. **Current source is not MSVC-x64 compatible.** `engine.cpp` uses GNU register variables and inline assembly; MSVC x64 has no inline assembler. CMake/README should not claim VS 2022 support without an alternate thunk.
3. **Five tests are gated on AngelScript source presence.** Lifecycle, hot reload, game-host integration, and float-global regression are nested with the benchmark under the AngelScript condition even though they do not require it. A checkout without the SDK silently drops them and reports 12 tests.
4. **CMake still declares project version `0.5.0`.** README and release docs call the project v1.0.
5. **The CLI does not link/register `sync`, despite README claiming all eight extensions.**
6. **CLI help omits implemented `--emit-em`, `--tick`, `--tick-count`, and `--tick-interval` options.** The positional `emit-em` action is parsed but does not set an output path, and documented `--load-em` is absent.
7. **Thread-safety evidence is narrower than claimed.** The two-thread portion of `thread_safety_test.cpp` compiles another module in thread B rather than invoking one shared compiled body with two contexts. Add the exact shared-body test promised by the integration notes.

## 6. Performance assessment

### 6.1 What the current benchmark proves

`bench_ember_vs_as` is reproducible: a fresh audit run was close to the committed ratios (approximately `0.13/0.17/0.17/0.52` versus `0.15/0.16/0.18/0.55`). It compiles once, checks result equality, and times the hot path with safety checks disabled. It validly demonstrates that baseline native Ember code is faster than the tested AngelScript bytecode interpreter on four workloads.

It does **not** establish that the tree-walking stack-spilling codegen is close to good native code or that SSA/linear-scan work is unnecessary. Native-versus-interpreted execution dominates the comparison. The harness also reports one mean over 5–10 iterations, has one warmup, no variance/confidence interval, no affinity control, and no machine/compiler metadata in the generated report. Running the CTest rewrites `v0.6_BENCHMARK_RESULTS.md` as a side effect.

### 6.2 Main hot-path costs found by inspection

1. Every script call emits a 10-byte `mov r11, imm64` dispatch-base load before indirect call; it is never hoisted or made RIP-relative.
2. Integer binary expressions push/pop the left operand because the tree walker has only one primary expression register.
3. Loop pinning holds exactly one local in `rbx`; accumulator+counter loops still spill one key value, and nested loops cannot acquire an inner pin.
4. Enabled budget/depth checks perform memory RMW operations at every loop back edge/script call.
5. Compile-time block handling copies unordered maps at every nested block; this affects compile latency, not script runtime.

These are optimization opportunities, not correctness findings. Safety checks should not be removed to improve benchmarks.

### 6.3 Better evidence plan

- retain the AngelScript comparison as an interpreter baseline;
- add a native C++/hand-native baseline for identical workloads;
- report min, median, standard deviation, and ratio confidence over at least 20 measured iterations after at least five warmups;
- record CPU, compiler, flags, date, affinity, and safety configuration;
- add call-only, two-local accumulator, expression-tree, struct-ABI, and safety-enabled workloads;
- write reports only on explicit request, not as a normal test side effect.

## 7. Documentation and test coverage gaps

### D1. Seven detailed docs still open with a stale “Implementation status: v0.1” banner

Affected files include `BINDING_API.md`, `COMPILER_PIPELINE.md`, `HOT_RELOAD.md`, `LIFECYCLE.md`, `MEMORY_AND_GC.md`, `SAFETY_AND_SANDBOX.md`, and `TYPE_SYSTEM.md`. Their bodies mix implemented v1.0 behavior and target design, so the blanket disclaimer is now false and obscures which sections actually shipped.

### D2. `GAP_ANALYSIS.md` claims linear-scan register allocation

The implementation is a tree-walking stack-spilling emitter; SSA-lite IR and linear-scan allocation remain deferred. This contradicts source, README, and `COMPILER_PIPELINE.md`.

### D3. Hot-reload documentation presents unimplemented mechanisms as shipped design

Epoch reclamation, whole-module reload, and removed-function trap stubs are specified but absent. The actual implementation supports single-function replacement and leaves old-page lifetime to a single-threaded caller.

### D4. Roadmap/gap docs overstate addon coverage

`map<K,V>` does not exist. Array lacks the documented general push/pop/remove/clear surface; string lacks documented find/substr/format natives. The docs should list the actual extension API rather than target names.

### D5. f-string documentation describes a nonexistent `__fmt` lowering

The parser creates `__fstring_to_string` segments and sema redirects them to type-specific `string_from_*`/identity conversions. No `__fmt` native exists. There is no positive f-string `.ember` test.

### D6. Positive execution tests are missing for claimed language features

Before this audit there were no positive runtime cases for `do-while`, prefix/postfix `++/--`, `sizeof`, `offsetof`, `constexpr`, and lexical `defer` behavior. That allowed several high/medium defects above to ship.

### D7. Most `valid_*.ember` files are parse-only

The language harness provides valuable classification coverage but does not execute most valid programs. `sema_valid_basics.ember` is explicitly skipped from sema because it references prism natives, despite broad README language about classifying the entire set.

### D8. `.em` tests cover only a friendly same-process subset

`em_roundtrip_test` avoids strings, natives, function references, complex globals, and cross-process loading. There are no malformed count/size/relocation, duplicate slot, signature mismatch, or fuzz tests.

### D9. ABI tests omit nonvolatile preservation and several aggregate shapes

Current tests cover many argument/return placements but not preservation of nonvolatile GP/XMM registers, >8-byte struct arguments, padded/nested structs, or recursive-layout rejection.

### D10. README benchmark wording rounds distinct ratios to the same value

The committed table reports `0.15`, `0.16`, and `0.18`; README calls all three approximately `0.15`. More importantly, the snapshot has no machine/statistical provenance.

### D11. Miscellaneous stale comments/spec details

- `binding_builder.hpp` still says `PERM_FFI` is not enforced, though sema enforcement shipped.
- `CODEGEN_SPEC.md` section numbering is out of order.
- `HOT_RELOAD.md` discusses relaxed atomics while the implementation correctly uses release/acquire.
- Some `BINDING_API.md` examples name APIs that do not exist.

## 8. Verified strengths

The following areas were explicitly checked and should not be reclassified as defects:

1. **W^X:** code pages are allocated RW, patched, changed to RX, then published. No normal RWX path was found.
2. **B1 context isolation:** compiled code obtains budgets/traps/allowlists through the per-caller `r14` context, and `--tick` uses its own `context_t`.
3. **Function-reference provenance guard:** indirect script calls perform unsigned range and allowlist-bit checks when the host supplies the guard metadata.
4. **`PERM_FFI`:** sema rejects calls to gated natives when module permissions omit the bit; the check has no runtime hot-path cost.
5. **Dispatch publication:** `DispatchTable` uses release stores/acquire loads.
6. **Registry base stability:** `ModuleRegistry` pre-sizes its entry storage and does not reallocate its base array.
7. **Kind-2 missing-registry handling:** a `.em` kind-2 relocation without a supplied registry is rejected rather than silently patched.
8. **Longjmp choice:** paired compiler builtins avoid unwinding through JIT frames without Windows unwind metadata.
9. **Parallel codegen metadata:** the current path prefers context-owned globals maps over the old process-global fallback.
10. **MPMC lock scope:** the sync extension does not hold its internal mutex across an Ember callback.

---

## 9. Trusted-host contracts and documented deferrals

These conditions are sharp edges but are not classified as vulnerabilities when their documented contracts hold:

- one `context_t` must not be entered concurrently by multiple threads;
- mutable script globals shared by contexts require host-provided synchronization;
- native bodies are trusted and may violate memory safety if implemented incorrectly;
- natives must obey the Win64 ABI, including preservation of `r14` during B1 calls;
- host-driven routine invocation is outside the JIT function-reference guard and must validate slots;
- bare `fn` signature erasure and cross-module function handles are documented deferrals;
- registry capacity/sentinel return values must be checked by the host;
- native slice APIs receive raw addresses by design, so `PERM_FFI` is a registration policy rather than full capability isolation;
- hot reload is safe with synchronous reclamation only when no concurrent caller can execute the old page;
- reload/compile mutation of the same `Program` must be serialized by the host.

The report recommends making these contracts prominent in the embedding API rather than relying on comments spread across examples.

## 10. Static-analysis candidates not yet reproduced

These require follow-up before being promoted to confirmed findings:

1. signed negative left shift in sema constant evaluation and codegen folding can invoke C++ undefined behavior;
2. signed overflow in sync atomic fetch-add/sub can invoke C++ undefined behavior even though hardware wrap is likely intended;
3. `X64Emitter::bind()` silently overwrites a duplicate label binding instead of reporting the internal compiler error required by the codegen specification.

## Conclusion

Ember's core architecture is coherent: one native execution path, atomic dispatch, W^X pages, per-thread contexts, compile-time FFI gating, and a real indirect-call provenance check. The main problem is not the architecture; it is that the accepted language and file-format surface is wider than the paths that currently receive runtime and adversarial testing.

The next release should prioritize boundary hardening and semantic truth over new features. After the `.em` loader, casts/handles/sizes, ABI preservation, and silently missing codegen paths are fixed—and after the test gate itself is deterministic—the existing design can support a substantially stronger v1.0 claim without adding speculative abstractions.
