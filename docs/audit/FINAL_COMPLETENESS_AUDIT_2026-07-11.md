## 0. Headline

Every feature the task asked about **shipped and is covered by at least one passing test.** The implementation work is done; the gap is **documentation drift**, not implementation gaps. Specifically:

- **16 of 17 features are COMPLETE** (full implementation + passing tests). The one PARTIAL feature is `iterable()` (slice + array-handle only, by design - the map/host-collection cases are explicitly deferred).
- **3 features have a test-coverage caveat**: GC has unit tests but is **not wired into the engine/codegen** (standalone core only); in-context threads has a dedicated ctest but **no lang tests and is not linked into `ember_cli`** (so `ember test`/`ember run` cannot exercise it); lambdas + coroutines are tested via `ember test` (ctest `ember_test_cli`) but **not in the bash `lang_suite`** (ctest `lang_suite`) explicit RUN list.
- **The ROADMAP Tier sections are STALE for 7 features**: GC, coroutines, exceptions, struct destructure, in-context threads, cross-module handles, and parameterized fn types all shipped this session but their Tier entries still say "TODO". The self-hosting narrative (lines 99-143) and the session summary (lines 144-200) only cover self-hosting + `.em` hardening - the other session features are not summarized as shipped anywhere in the ROADMAP.
- **The user guide (`docs/guide/`) documents NONE of the session features.** constexpr, static_assert, typed enums, lambdas, coroutines, try/catch/throw, namespaces, struct destructure, fn(Args)->Ret, cross-module handles, in-context threads, GC, iterable, and slice-escape are all absent from `docs/guide/10-language/`. The guide covers only the pre-session surface.
- **Two spec docs have stale "v2+/non-goal" claims** that contradict shipped code: `TYPE_SYSTEM.md` S8 ("parameterized `fn(i64)->i64` types are v2+. Closures (lambdas with capture) remain a non-goal for v1") and `MEMORY_AND_GC.md` S8 ("v2 GC deferral ... No feature in the v1 language surface creates that situation").
- **There are uncommitted working-tree edits** to `README.md` and `extensions/README.md` that partially close the doc gap (they add pipe/live CLI lines, thread/coroutine/call_raw extension mentions, and the regalloc paragraph). These edits are NOT committed at audit time. They address README + the extensions CMakeLists paragraph, but the extensions README **table** still lacks rows for `coroutine/`, `thread/`, and `call_raw/`.

The matrix below is the feature-by-feature verdict. Section 2 is the per-feature evidence. Section 3 is the documentation-drift list. Section 4 is the test-coverage caveats. Section 5 is the recommended follow-ups.

## 4. Test-Coverage Caveats

### 4.1 GC - unit tests only, not integration-tested

`gc_test.cpp` (ctest `gc_core`) exercises the `GcHeap` directly (alloc/collect/root scanning/cycles/stats). There is NO test that JIT'd code calls `gc_alloc`/`gc_collect`, because the JIT does not emit those calls (the GC is not wired into `engine.cpp`/`codegen.cpp`/`context.hpp`). This is correct for the current state (the GC is foundational infra shipped ahead of its consumers) but means the GC's integration with JIT frame root scanning is untested. When the GC is wired in (for lambdas-with-by-reference-capture or coroutine suspended-frame heap storage), an integration test is needed.

### 4.2 In-context threads - no lang tests, not in `ember_cli`

`ember_ext_thread` is linked into `in_context_threads_test` only - NOT into `ember_cli` (the `ember_cli` link list at `CMakeLists.txt:521` includes `ember_ext_sync` but not `ember_ext_thread`). Consequence: `ember test` and `ember run` cannot use `thread_spawn` (the native is not registered). There are no `tests/lang/valid_thread*.ember` files. The feature is covered by the dedicated ctest (`in_context_threads`, 4 scenarios) + `thread_safety` (the per-context two-thread proof), so it IS tested - but a script author cannot write `thread_spawn(...)` in a `.ember` file and run it via the CLI. If the CLI should expose in-context threads, add `ember_ext_thread` to the `ember_cli` link list + `ext_thread::register_natives` to the CLI's native-registration block + `thread_init` call (mirroring `coroutine_init`).

### 4.3 Lambdas + coroutines - in `ember test` but not in `lang_suite` bash explicit RUN list

The bash `lang_suite` (ctest `lang_suite`, `tests/run_lang_tests.sh`) has an explicit RUN list (the `for spec in ...` loop at lines 87-113) that names `valid_constexpr*`, `valid_static_assert*`, `valid_typed_enum*`, `valid_fn_types`, `valid_namespaces*`, `valid_struct_destructure`, `valid_match_guards`, and all `valid_try_*`/`valid_throw_*` files - but NOT `valid_lambda*` or `valid_coroutine*`. Those 9 files ARE run by `ember test` (ctest `ember_test_cli`, which auto-discovers every `.ember` with `// expect:`), so they ARE covered - but the two ctest targets have different coverage. The `lang_suite` is the stricter gate (it asserts exact exit codes); `ember test` also asserts exact exit codes via `// expect:`. So the coverage is equivalent in strength, just split across two runners. If the `lang_suite` should be the single source of truth, add the 9 lambda/coroutine specs to its explicit list.

### 4.4 Self-hosting - not in ctest

The 5 self-hosted tests run manually via `ember_cli.exe run self_hosted/<stage>_test.ember --fn main` (the pipeline needs `--ffi`). They are NOT registered as ctest targets. The ROADMAP documents the manual invocation. A regression could land without being caught by `ctest`. Wiring them into ctest (as 5 `add_test` invocations that call `ember_cli run self_hosted/<stage>_test.ember --fn main` with the appropriate TIMEOUT) would close this.
