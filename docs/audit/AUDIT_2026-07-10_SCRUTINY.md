### M4 budget test — concrete verdict

The M4 budget tests live in `examples/v0_4_hardening_test.cpp` (a MODIFIED, not NEW, file). There are two budget-relevant assertions:

1. **Runaway loop (`:267-273`), budget=1000.** `run_under_safetrap("fn main() -> i64 { while (true) { let x: i64 = 1+1+1+1+1; } return 0; }\n", 1000)` expects `BudgetExceeded`. Body is one `LetStmt`; `block_cost(body) = 2` (1 stmt-overhead + 1 stmt_cost); the back-edge charge (`codegen.cpp:2330`) subtracts 2 per iteration, so ~500 iterations exhaust budget=1000. **This budget is NOT tautologically small.** A legitimate bounded loop of ≤500 iterations completes under budget=1000; only an *infinite* `while(true)` (no break) traps. The defect the M4 budget exists to catch — the 07-09 M4 finding "long straight-line and call-heavy execution can avoid instruction accounting" → infinite hang — is exactly what this tests. The 07-09 finding's reproducer was a hang; this test asserts the hang is gone and replaced by a recoverable trap. **REAL.** Caveat: this test was NOT added by the remediation — `git show 381ed76:examples/v0_4_hardening_test.cpp` and `git show 863284e:...` both contain it (line 177/270 respectively). It pre-dates the remediation, so it is not evidence of remediation coverage; it is evidence the pre-existing loop-back-edge charge works. The audit does not claim it as new evidence, so no correction is needed beyond noting its pre-existence.

2. **Straight-line entry-charge (`:276-280`), budget=3.** `run_under_safetrap("fn main() -> i64 { let a=1; let b=2; let c=3; let d=4; return a+b+c+d; }\n", 3)` expects `BudgetExceeded`. Body = 5 leaf statements (4 Let + 1 Return) → `block_cost = 2×5 = 10`. The M4 entry charge (`codegen.cpp:2688`, added by pass 2) emits `emit_budget_check(10, "budget exceeded at function entry")`; budget=3 < 10 → traps immediately at entry. **This test is NOT tautologically small and NOT circular.** A budget of 3 is calibrated: a minimal 1-statement `fn main()->i64 { return 0; }` has `block_cost = 2` (≤ budget 3) and would *complete*; the 5-statement body (cost 10) traps. So the test discriminates "entry charge scales with statement count." Crucially, **passing it genuinely requires the entry-charge fix**: `git show 381ed76:src/codegen.cpp | grep 'function entry'` and `git show 863284e:src/codegen.cpp | grep 'function entry'` both return *nothing* — the entry charge did not exist before `baa90c1`. ### Q3. Is cross-machine or different-compiler v2 compatibility REAL or ASSUMED?

**Probe.** Only one compiler is available on this machine: MinGW g++ 15.2.0. `which clang clang++ cl` → all absent; `/c/msys64/mingw64/bin/clang*` and `/c/msys64/clang64/bin/clang*` → no such file. So a true cross-compiler *acceptance* test (emit under g++, load under clang/MSVC, expect success) is **UNVERIFIABLE on this single machine**.

**What the code does (mechanism verification):** `EM_BUILD_ID` **does encode compiler identity** — `em_file.hpp:46-56` has four `#if defined(__clang__)/__GNUC__/_MSC_VER/else` branches producing distinct hashes (`"ember-em-v2;clang=…"`, `"…;gcc=…"`, `"…;msvc=…"`, `"…;compiler=unknown"`). `EM_TARGET_ABI_HASH` encodes calling convention (`EMBER_EM_CC`: `win64-x64`/`sysv-x64`/`aapcs64`) and arch/ptr/os. So a g++-emitted artifact read by a clang-built loader would mismatch on the build-id (gcc≠clang) AND on the ABI hash (both win64-x64 on Windows x64, so the ABI hash might match but the build-id would not), and the loader rejects at `em_loader.cpp:219`: `if(build!=EM_BUILD_ID||reserved[2]!=EM_TARGET_ABI_HASH){set_error(err,"em_loader: compatibility: compiler/build ID or target ABI hash mismatch");return false;}`.

**Reject-path empirical confirmation:** I corrupted the build-id byte at offset 28 in a copy of `/tmp/probe_b1.em` (`printf '\x00' | dd … seek=28 … conv=notrunc`) and ran `buildt/ember_cli.exe run --load-em /tmp/probe_corrupt.em --fn main` → `ember: load failed: em_loader: compatibility: compiler/build ID or target ABI hash mismatch`, rc=2. Same for a corrupted ABI hash at offset 36. Both reject **before any page publication** (the loader's parse_file validates identity at `:216-219` before the allocation loop at `:280+`).

**What a real cross-compiler test WOULD require:** (a) a second compiler (clang-18 or MSVC 19.4x) installed; (b) emit a v2 artifact from a g++-built `ember_cli.exe`; (c) load it from a clang-/MSVC-built `ember_cli.exe`; (d) assert the load **fails** with the `:219` mismatch (different build-id) — this is the *intended* behavior, since native x64 code is not ABI-portable across compilers with different `__int128`/struct-return/`long double` layouts, and the format's design is to **reject** cross-compiler artifacts rather than mis-execute them. A test that expected cross-compiler *acceptance* would be testing for a bug, not a feature. The code's ABI-ID **does** encode compiler identity (the necessary precondition for cross-compiler rejection to be sound), at `em_file.hpp:46-56`.

**Verdict: UNVERIFIABLE-on-single-machine** for true cross-compiler accept/reject (no second compiler present); **the mechanism is REAL and partially verified** — the build-id encodes compiler identity (`:46-56`), the loader rejects identity mismatch (`:219`), and the reject path fires correctly under both build-id and ABI-hash corruption. The cross-compiler *rejection* is by-design (native code is not cross-compiler portable); the audit does not over-claim cross-compiler compatibility — it lists it in §1 as an explicit deferral.

**Correction to Audit:** none. The audit's §1 explicitly defers "cross-machine/cross-compiler v2 compatibility" and its §5 H12 strength claims only "separately launched processes built from the same compiler artifact agree" — not cross-compiler agreement. The claim is correctly scoped. (If the audit had claimed cross-compiler portability, it would be an overstatement; it does not.)

### Q4. Are ALL canonical-Type signature dimensions combinatorially checked by `canonical_type_same`, and is each dimension mutation-tested?

**Probe (comparison completeness).** `EmSignature` = `{Type ret; std::vector<Type> params;}` (`em_file.hpp:188-191`). `Type` (`ast.hpp:20-39`) has 9 structural fields: `prim`, `struct_name`, `is_slice`, `array_len`, `elem` (shared_ptr, recursive), `is_fn_handle`, `has_recorded_sig`, `recorded_params` (vector, recursive), `recorded_ret` (shared_ptr, recursive). `canonical_type_same` (`em_loader.cpp:175-181`) compares **all 9**: `prim`/`struct_name`/`is_slice`/`array_len`/`is_fn_handle`/`has_recorded_sig` (line 176), `elem` recursively (177), `recorded_params.size()`/`recorded_ret` existence (178), each `recorded_params[i]` recursively (179), `recorded_ret` recursively (180). **The comparison is complete over every structural dimension a `Type` possesses.** Ember's `Type` has **no calling-convention, constness, or variadic field** (`ast.hpp:20-39` confirms), so there is no missing dimension in the comparison. `NativeSig::permission` (`sema.hpp:42`) is **not** part of `EmSignature` (it's a registration policy, not ABI) and is correctly **not** compared (`grep permission src/em_loader.cpp` → 0 hits) — comparing it would be a defect.

**Probe (test coverage of each dimension).** The loader's native-binding signature check (`em_loader.cpp:345-347`) compares `it->second.ret` vs `b.signature.ret` and each `it->second.params[pi]` vs `b.signature.params[pi]`. The only test that mutation-exercises this check at the **loader boundary** is `em_roundtrip_test.cpp` [4c] (`:268-276`): it builds a v2 module with `host_bound: i64->i64`, then loads it against a wrong allowlist with `host_bound: f32->i64` and asserts rejection with "signature mismatch." This exercises **exactly one dimension** — the return-type `prim` (I64 vs F32). No test mutates `is_slice` vs `array_len`, `struct_name`, `is_fn_handle`, `has_recorded_sig`, `recorded_params`, or `recorded_ret` at the loader `canonical_type_same` boundary to confirm each dimension is individually checked. The `em_loader_hardening_test.cpp` suite covers **v1 bounds** (huge_code, reloc_wrap, duplicate_slot, trailing, etc. — 23 cases at `:82-104`) but **zero canonical-type shape or dimension cases** (`grep -n 'Ghost\|Void.*struct\|inconsistent\|is_slice\|is_fn_handle' examples/em_loader_hardening_test.cpp` → 0 hits). The audit's own M-H14-1 probe (`tmp_edit/c4/h12_h14_probe.cpp malformed`) covers one shape case (Prim::Void + nonempty struct_name accepted) but no per-dimension comparison-rejection cases.

**Probe (shape validation completeness — relates to M-H14-1).** `parse_type` (`em_loader.cpp:160-172`) validates only 5 shape invariants: nesting depth>16 (`:162`), `prim>F64` (`:165`), `flags&~31` (`:165`), `name_len>MAX_NAME_SIZE` (`:165`), and `is_slice && array_len` (both slice AND array, `:169`). The writer (`em_writer.cpp:91-92`) **derives** the struct flag from `!struct_name.empty()` and writes it, but the loader **ignores bit 2** on read (`:167` reads bits 0,3,16 → is_slice/is_fn_handle/has_recorded_sig; bit 2 is never consulted), so the on-disk struct flag is informational only and cannot enforce consistency.

**Verdict: DEFECT (test-coverage gap, not comparison defect).** The `canonical_type_same` comparison itself is **complete** over all 9 `Type` dimensions — no dimension is missing from the comparison. The defect is the **absence of mutation tests per dimension**: only the return-`prim` dimension is exercised at the loader boundary (`em_roundtrip_test.cpp:274`); the other 8 dimensions (slice-vs-array, struct_name, fn_handle, recorded_sig, recorded_params count, each recorded_param, recorded_ret, elem recursion) have no test confirming `canonical_type_same` actually rejects their mismatch. Additionally, `parse_type`'s shape validation is incomplete (5 of ~9 invariants enforced) — this is the **same gap the audit's M-H14-1 already flags**, and the audit's proposed fix ("struct flag iff name nonempty; fn flags iff Fn; recorded signature implies fn; array flag iff nonzero length; slice/array require one valid element; reject impossible primitive/flag combinations recursively") is **correct and complete** for the shape-validation half. The per-dimension comparison-rejection tests are a **new gap** the audit's M-H14-1 fix description mentions ("with mutation tests for each impossible combination") but which the shipped test suite does not contain.

**Correction to Audit:** the audit's M-H14-1 finding is accurate and its fix is sound+complete, but it is **classified too narrowly** — it is filed only as a *shape-validation* defect ("internally inconsistent canonical type metadata").
