# ember v1.0 integration notes — the concurrency + Tier 2 batch

The v1.0 batch (commit `e5d1814`, "Concurrency batch: context thread-safety +
enums + sync queues + first-class fn refs") is the follow-on to the v0.6
integration. It ships four features, every claim verified against the source
and tests (the four `docs/plan_*.md` files are the historical plans, left
as-is). Three follow-on commits then built the dynamic-registration extension
the Tier 2 fn-refs feature enables and wired the CLI `--tick` path to the
B1 model (see §5). The current tree configures 20 CTest tests with the
AngelScript SDK and 19 without it; only the comparison benchmark is
conditional. The batch originally added `thread_safety`, `function_refs`,
`ext_sync`, and `ext_lifecycle`, with later audit regressions expanding the gate.

## 1. Context thread-safety (Option D + B1)

The v0.4 non-local abort primitive (`../spec/SAFETY_AND_SANDBOX.md` §2-§4) was
single-threaded-by-assumption: one checkpoint, one budget counter, one depth
counter, all on one `context_t`. v1.0 makes the per-`ember_call` state
**per-thread** so one compiled module can serve N concurrent caller threads,
each with its own `context_t`.

- **Option D — one `context_t` per thread.** `context_t` (`src/context.hpp`) is
  restructured to a **POD prefix**: the JIT-read fields (`budget_remaining`,
  `call_depth`, `max_call_depth`, then `last_trap`) come first and are POD, so
  `[ctx_reg + offsetof(field)]` is standard and stable across compilers (the
  trailing `std::string last_error` is non-POD and stays out of the offset
  math). `context_offsets()` gives the exact byte offsets the codegen bakes
  into `[r14 + off]`. Each concurrent caller thread allocates its own
  `context_t` (private checkpoint/budget/depth); they share the dispatch
  table, JIT'd code, and module registry, all read-only after compile. A trap
  on thread B longjmps to thread B's own checkpoint — no cross-thread
  checkpoint corruption.
- **Option B1 — the per-call context register.** With
  `CodeGenCtx::use_context_reg = true`, the budget/depth/trap emits
  (`emit_budget_check`, `emit_depth_check`, `emit_depth_leave`, `emit_trap` in
  `src/codegen.cpp`) read `context_t` fields through `r14` (the per-call context
  register, callee-saved so a script-to-script call forwards it — the callee
  inherits the same ctx) instead of baked imm64 pointers. The trap stub's first
  arg (`rcx`) is loaded from `r14`. The load-bearing consequence: **one
  compiled body serves N per-thread `context_t`s** — no per-context recompile.
  The host sets `r14 = ctx` at entry via the `ember_call_void` /
  `ember_call_i64` inline-asm helpers (`src/engine.{hpp,cpp}`); legacy non-B1
  hosts keep the baked-pointer behavior (`use_context_reg = false` is the
  default, backward compatible).

Keystone proof: `examples/thread_safety_test.cpp` (ctest target
`thread_safety`) — two host threads run the same compiled entry with two
separate `context_t`s (different budgets); thread A finishes and returns,
thread B traps `BudgetExceeded` on its own `context_t`, and neither thread's
longjmp corrupts the other.

**The emit-time globals fix that came with it.** Codegen now prefers
`ctx.globals_index` / `ctx.globals_types` (threaded through `CodeGenCtx`,
`src/codegen.hpp` + `src/codegen.cpp`) over the legacy process-wide
`g_globals_for_codegen` pointer — the process-wide pointer races under
parallel compilation (two threads compiling into the same engine
simultaneously would both read/write it). The legacy pointer stays as a
null-by-default backward-compat fallback so existing hosts keep working; a
B1 host threads the index/types through `CodeGenCtx` and leaves
`g_globals_for_codegen = nullptr` (the `thread_safety_test` proves this path:
it nulls the pointer before compile and reads a global through
`ctx.globals_index`).

### The `--tick` bug — now fixed (commit c6457cd)

The v0.6 `--tick` lifecycle mode (`ember_cli --tick`, a tick thread + TUI
unload keybind) had a latent bug: the tick thread reused the main thread's
`context_t`, so a budget/overflow trap in a tick could longjmp to the MAIN
thread's checkpoint, corrupting its state (the generalized S1.4 `--tick` bug).
That is exactly the bug B1 + per-thread `context_t` fixes. **The CLI `--tick`
path is now a B1 host** (commit `c6457cd`): the CLI compiles with
`use_context_reg = true`, calls `@entry` via `ember_call_void(entry, &ectx)`,
and the tick thread uses its **own** `context_t` (`tick_ctx`) via
`ember_call_void(f, &tick_ctx)`, fully isolated from the main thread's
`ectx` — a tick trap stops the tick thread (sets `tick_trapped`), never the
main thread. The same commit wired the v1.0 parallel-compile-safe globals
(`ctx.globals_index`/`ctx.globals_types`, `g_globals_for_codegen` now null in
the CLI) and the fn allowlist (`build_fn_allowlist`) so a CLI-loaded script
that uses `&fn` / `handle(args)` works. Verified: `--tick-count 5` runs 5
ticks cleanly; a tick that budget-traps (infinite loop) is caught on the tick
thread, the main thread survives, exit 0. (The `examples/thread_safety_test.cpp`
keystone proof from §1 covers the underlying B1 invariant; the CLI wiring is
the first host to consume it.)

## 2. Enums (Tier 1)

Script-side `enum E { A, B, C }` + `E::A` qualified access. Top-level decl in
`src/parser.cpp` (`parse_enum`); sema's `resolve_enums` pass resolves each
variant's value (auto-increment from 0, or from the last explicit
`= constexpr_int_expr`, via the existing `try_eval_const_i64`) and
`lower_enum_access_expr` rewrites the `EnumAccessExpr` to an `IntLit` **in
place**, so by the time `check_expr` runs there are no `EnumAccessExpr` nodes
left — codegen, the const-folder, the switch case-value literal check, and the
globals initializer evaluator all see an ordinary integer literal. **No
codegen change** (codegen's `eval()` has no `EnumAccessExpr` case and never
needs one).

Variants are **i32 values** (the `enum_values` table is `int64_t` but the
default `IntLit` type is `Prim::I32`, sign-extended into the `IntLit`'s i64 `v`
field); usable anywhere an `i64` is, exactly like an untyped integer literal.
An enum name is **not** a type in v1 — `let x: Color = ...` is a clean sema
error (the `check_type_not_enum` hook typed enums later flip to accept).
Duplicate enum names / duplicate variant names within one enum / duplicate
explicit values are errors (stricter than C).

Pinned by `tests/lang/{valid_enums,sema_valid_enums,
sema_invalid_enum_unknown_enum,sema_invalid_enum_unknown_variant}.ember`
(four files; the v1.0 batch added them).

## 3. Sync queue primitives (`extensions/sync/`)

A new extension (the seventh in the batch; `lifecycle` made it eight in a
follow-on commit). Atomics (`aint8/16/32/64`: load/store/
fetch_add/fetch_sub/cas/swap with width masking), a swap buffer
(double-buffer + atomic flip), and SPSC/MPSC/MPMC queues — all behind opaque
i64 handles, with internally synchronized host storage (`std::atomic` /
lock-free ring / host-internal `std::mutex` for MPMC held only across a short
ring-index critical section, never across an ember call). 1-based handles
with a free-list; `reset()` clears all five stores (mirrors `ext_array`/
`ext_string`).

**The U2 contract** (`ext_sync.hpp`'s S0 scope statement — read it first):
the script side stays single-threaded per context; the host threads that
produce/consume touch only the queue/swap-buffer/atomic HOST storage via the
`*_host` accessors, never the context. The queues work because the contention
is on host storage, not on the context. **These primitives do not make a
`context_t` safe for concurrent calls** — two ember-calling threads on one
`context_t` still race exactly as `plan_CONTEXT_THREADSAFETY.md` S1.2 documents
(the S1.4 `--tick` bug, generalized). The sync addon is usable the moment the
host arranges that only one thread at a time calls ember into a given context
— which is the model §1's Option D+B1 makes ergonomic.

All queue natives are **non-blocking** (push returns 0 on full, `try_pop`
returns `INT64_MIN` on empty); a script cannot deadlock on any primitive here
(the only "blocking" is a script-authored poll loop, which the instruction
budget bounds, `../spec/SAFETY_AND_SANDBOX.md` §3).

Pinned by `examples/ext_sync_test.cpp` (ctest target `ext_sync`): 16 tests
incl. multi-thread stress (10k SPSC, MPMC contention, no lost/dup), the
free-list reuse / store-grows-under-leak / reset-reclaims lifecycle, and
atomic cross-thread signal (host store, main poll via the load native).

## 4. First-class function refs (Tier 2)

`&fn` takes a handle, `handle(args)` calls indirectly, `fn` is the handle type
keyword. Parser (`src/parser.cpp`): prefix `&` in `parse_unary` builds a
`FnHandleExpr`; a relaxed call target (`<expr>(args)` where `<expr>` is none of
the named forms) builds an indirect `CallExpr` with `indirect_target` set;
`parse_type` treats bare `fn` as `Prim::I64` with `is_fn_handle=true`. Sema
(`src/sema.cpp`): `FnHandleExpr` bakes the slot as an i64 literal (and records
the source fn's signature on the type for arg checking); the indirect
`CallExpr` path type-checks args against the recorded signature when present; a
local fn-typed var called like a named fn is promoted to the indirect path;
**i64 ↔ fn assignment is forbidden either direction** (closes V3 forging at
the type level).

**The call-target-provenance guard.** `emit_call_target_guard` in
`src/codegen.cpp` validates the runtime i64 against a host-built bitset
allowlist (range check + `bt` bit test) **before** indexing the dispatch
table; a bad handle traps `BadCallTarget` (longjmp to the `ember_call`
checkpoint), never a raw `call rax` / `call [table + h*8]` with an unvalidated
script-supplied i64. `build_fn_allowlist` (`src/codegen.hpp`) is the host
helper that derives the bitset from `script_slots`; the host pins its
`.data()` base for the module's lifetime and sets `CodeGenCtx::fn_allowlist_base`
/ `fn_slot_count` before compiling. The guard is a **no-op (zero emitted)**
when no allowlist is configured (`fn_slot_count == 0`), so existing modules
that don't use function refs pay nothing. `BadCallTarget` is a `TrapReason`
(`src/context.hpp`), routed through the same shared unwind as every other
trap (`../spec/SAFETY_AND_SANDBOX.md` §7a has the full invariant).

Pinned by `examples/function_refs_test.cpp` (ctest target `function_refs`):
handle creation, multi-arg dispatch (Win64 `rcx`/`rdx`), recursion via handle
(`fib` via `&fib`), the type-level forging rejections (`let f: fn = forged_i64`
rejected both directions), and the guard (out-of-range and
in-range-unregistered handles both trap `BadCallTarget`).

### Two open items (from `plan_FUNCTION_REFS.md` §9.3 / §9.5)

- **The bare-`fn` signature hole.** A `fn`-typed *parameter* (`fn h`, with no
  recorded signature) accepts any args at the call site — sema does not
  type-check them, the runtime guard still validates the handle is a
  registered slot before dispatch, and the called code obeys all
  budgets/bounds. This is a **type-soundness hole, not a sandbox violation**
  (a `let f: fn = &add; f(1);` call dispatches into `add` with `rcx=1,
  rdx=garbage` and returns a garbage result with no sema error). The fix is
  `fn(i64)->i64` parameterized function types (signature equality,
  parameterized types) — a real type-system expansion, deferred to v2+. The
  guard ensures the *call target* is always safe; the *args* are the caller's
  responsibility at the bare-`fn` type, same as a C function pointer with no
  prototype. (The shipped `register_routine(fn h, i64 data)` native in §5
  below is itself a bare-`fn`-typed param — the host trusts the slot, calls it
  with the `data` arg the script supplied, same as the static `@on_tick` path.)
- **Cross-module function handles — deferred to v2+.** `&mod::fn` / `mod::fn`
  returning a handle is not in scope: the allowlist is per-module (built
  from this module's `script_slots`), and a handle from another module is
  that module's slot index, meaningless against this module's dispatch table.
  Cross-module handles need either a global `(module_id, slot)` handle space
  or per-module allowlists with a guard that knows which module a handle is
  for. Tier 2 ships intra-module handles only.

Both are documented at `../ROADMAP.md` Tier 2 with their re-entry triggers.

## 5. Dynamic routine registration — `extensions/lifecycle/` (follow-on commits)

The Tier 2 function-ref feature enabled the dynamic-registration path
`../LIFECYCLE.md` §2 flagged as needing function refs. Three follow-on commits
built it:
- `6d4dedb` — `extensions/lifecycle/` (`ext_lifecycle.{hpp,cpp}`):
  `register_routine(fn h, i64 data) -> id` / `unregister_routine(id)`. The
  `fn` param is typed (`is_fn_handle`) so sema rejects a forged plain i64;
  the slot's provenance was validated at the `&fn` site, so the host trusts it
  the way it trusts any sema-resolved slot and calls it via the dispatch
  table — the SAME call mechanism as the static `@on_tick` path, just
  discovered by the script at runtime. The host's `ext_lifecycle::
  host_routines()` accessor returns the `(slot, data)` pairs for the host to
  iterate + call per frame. 1-based ids, tombstone + free-list, `reset()`
  clears (mirrors `ext_sync`). Pinned by `examples/ext_lifecycle_test.cpp`
  (ctest target `ext_lifecycle`): full round-trip (register →
  host_routines → host calls via dispatch → unregister → free-list reuse →
  reset).
- `e6463b0` — wired the CLI `--tick` loop to drive dynamically-registered
  routines (the host ticks the script's registered routines each frame,
  alongside the static `@on_tick` fns).
- `a5a72f1` — `examples/scripts/dynamic_registration.ember`, the demo
  script (run: `ember run dynamic_registration.ember --tick --tick-count 5
  --tick-interval 50`).

This is the `register_routine`-shaped host native the original `../ROADMAP.md`
Tier 2 entry described as "trigger-gated on a real mod needing runtime-
decided callbacks" — the v0.6 integration + the Tier 2 fn-refs batch together
fired that trigger, so the reference native shipped in-tree (as an extension,
not a core change).
