# ember v0.6 integration notes — the first real embedding-target integration

The v1.0 example game-engine integration (`examples/game_host.cpp` +
`examples/scripts/game_logic.ember`) is the first end-to-end proof that ember
works as a game-logic embedding target. It exercises every host integration seam:
`@entry` setup, `@on_tick` per-frame over host state, `@event("player_input")`
response, native bindings operating on host-owned entity state, and hot reload
of a gameplay fn mid-loop. 12/12 ctest green.

## What it proved (the embeddability seams)

- **Lifecycle** (v0.6 `lifecycle.hpp`): `get_entry_function` + `get_annotated_functions("@on_tick")` + `get_event_handlers("player_input")` discover the script's entry/tick/event fns; the host calls them via the dispatch table. Works in a real host, not just a test harness.
- **Native bindings** (v0.3 `BindingBuilder`): the host registers `get_entity_count`/`get/set_entity_x`/`get/set_speed`/`get_delta_time`/`log` operating on a host-side `std::vector<Entity>` + a host float. The script reads/writes host state through these — the i64-handle/slice convention the six extensions already use. **No fluent TypeBuilder/StructBuilder/engine_t needed.**
- **Hot reload** (v0.6 `hot_reload.hpp`): mid-loop, the host reloads the `update` fn to move entities LEFT instead of right; the dispatch slot swaps and the next 5 frames use the new body. Per-function independence holds in a real host context.
- **`--tick` lifecycle mode** (v0.6 `ember_cli --tick`): a separate proof that the tick lifecycle works on the terminal runtime, shared shape with this host.

## The v1.0 fluent-API decision (the integration was the trigger test)

The v0.3 deferred-binding analysis (`v0.3_DEFERRED_BINDING_ANALYSIS.md`)
concluded the fluent `TypeBuilder`/`StructBuilder`/`engine_t` surface is
trigger-gated on "a host wants script-visible C++ struct types." This
integration is the first real host. **It used BindingBuilder + i64 handles +
StructLayoutTable for all natives and hit NO binding friction.** The trigger
remains unfired. The fluent API stays deferred to v1.0+ — now with evidence
from the first real integration, not just the extension audit.

If a future host wants to expose a C++ struct with fields the script reads/
writes directly (not through get/set natives), THAT fires the trigger — and the
use case will shape the TypeBuilder API. Until then, the i64-handle + native-
access pattern is sufficient (it's what AngelScript's scriptdictionary and the
surveyed native-JIT language's addons do too).

## A real codegen bug the integration found — now FIXED

The integration exposed a codegen bug: **float global WRITES were broken.** A
script global of type `f32`, written via `g = <float expr>`, did not land in
the globals block in the real (heap-allocated GlobalsBlock) host case.

**Root cause** (two parts):
1. `codegen.cpp`'s AssignExpr global-store path called `store_rax_to_global`
   (stores RAX, a 64-bit GP store) even for float globals — but a float value
   is in **xmm0**, not rax. So `g = 10.0f` stored whatever was in rax (garbage).
2. The `store_xmm0_to_global` fix I added had a **REX-prefix-ordering bug**:
   `movss [r11+disp32], xmm0` needs REX.B to extend rm=r11, and REX must come
   LAST (immediately before the opcode `0F 11`), AFTER the mandatory `F3`
   prefix. I emitted `41 F3 0F 11` (REX before F3) — the decoder drops a REX
   before a legacy prefix, so it became `movss [rbx+disp32]` (rbx, not r11) →
   stored to the wrong address. The stack-GlobalsBlock case happened to work
   because rbx held a coincidentally-writable address; the heap case segfaulted.

**Fix:** `store_xmm0_to_global` emits `F3 41 0F 11 83 <disp32>` (F3 prefix,
then REX.B, then opcode) — correct prefix order, r11 correctly addressed.
Verified by `examples/float_global_regression.cpp` (CTest target
`float_global_regression`): 5 cases — heap-GlobalsBlock write, read-modify-
write, two f32 globals, i64 regression (int path unbroken), f32 global read in
an expression. 13/13 ctest green.

The game_host integration keeps the host-owned-speed pattern (get/set_speed
natives) because it's the idiomatic game-scripting shape (host owns
authoritative state; the script reads/writes it through natives) — not because
the float-global path is still broken. Both patterns now work.

- **Also found (now FIXED):** ember did not evaluate global *initializers*
  at load (the globals block was zero-initialized; `global g : f32 = 10.0f`
  started at 0, not 10). Fixed: `src/globals.hpp` `eval_global_initializers`
  folds const initializers (literals + constexpr-foldable exprs like `6*7`)
  at load and writes them into the globals block. The CLI, game_host, and the
  float_global_regression test all call it after sizing the block. Non-const
  inits (`g = some_fn();`) still stay zero (the fns don't exist at load) — a
  host or `@entry` seeds those. Pinned by float_global_regression cases 6-9.

## Follow-on: the `--tick` thread-safety bug (resolved in v1.0)

The v0.6 `--tick` tick-thread mode reused the main thread's `context_t`, so a
budget/overflow trap in a tick could longjmp to the MAIN thread's checkpoint,
corrupting its state (the generalized S1.4 `--tick` bug). This file's prose
("suggested follow-on") described the bug as fixable; v1.0 actually fixed it.
The v1.0 context thread-safety work (Option D + B1: per-thread `context_t`,
the `r14` per-call context register) made one compiled body serve N per-thread
contexts, and commit `c6457cd` then wired the CLI `--tick` to compile with
`use_context_reg = true` and run the tick thread on its **own** `context_t`
(`tick_ctx`) via `ember_call_void(f, &tick_ctx)`, fully isolated from the main
thread's `ectx` — a tick trap stops the tick thread, never the main thread. See
`v1.0_INTEGRATION_NOTES.md` §1 for the full mechanism + the keystone proof
(`examples/thread_safety_test.cpp`), and §5 for the dynamic-registration
follow-on that the v0.6 `@on_tick` runtime-effect work and the Tier 2 fn-refs
batch together enabled.
