# ember v0.3 deferred binding surface — analysis

> **Historical analysis; revalidated 2026-07-15.** `BindingBuilder` remains the
> production API and `PERM_FFI` remains compile-time enforced. One major
> predicted gap did close: `register_struct` + `bind_struct` now register
> C++-layout first-class by-value host structs without requiring the proposed
> fluent `TypeBuilder`; `host_struct` pins this path. Slice-retention metadata
> also ships on `BindingBuilder::add`. The full fluent
> `TypeBuilder`/`StructBuilder`/`engine_t` and C-friendly descriptor ABI remain
> deferred because current hosts/extensions still do not require them. Enums
> are now untyped+typed source features, but host `EnumBuilder` remains absent.
> Counts of “six extensions” below are historical; current runtime extension
> directories number 15, plus opt/obf.

Research analysis of the host-facing binding surface that `../spec/BINDING_API.md`
specifies as the v1.0 ergonomic target and that v0.3 deliberately does **not**
ship. Scope: `TypeBuilder`, `StructBuilder`, `EnumBuilder`, `engine_t`,
`NativeFn` / `NativeParam` / `TypeId`, and full permission gating in codegen.

This is analytical research, not advocacy. It reads the spec against the
actual shipped code (the six `extensions/*/ext_*.cpp`, the three host exes
`examples/{ember_cli,sema_check,ext_registration_test}.cpp`, the two
round-trip tests, `src/{sema.hpp,sema.cpp,ast.hpp,binding_builder.hpp,
codegen.cpp,codegen.hpp}`) and asks, per deferred item: what it is, what it
would take, the *real* re-entry trigger grounded in the extension code (not
the roadmap's generic trigger), and an honest gap assessment. A final section
gives a recommended v1.0 binding surface and build order.

Prior art grounding is at the bottom: AngelScript's real registration surface
(decl-string + `RegisterGlobalFunction` / `RegisterObjectType` /
`RegisterObjectMethod` / `RegisterObjectProperty`) and Mun's binding model
(`#[repr(C)]` `FunctionDefinition`/`FunctionSignature`/`TypeId` ABI carried by
the AOT-compiled `.munlib`, host *discovers* `extern fn` declarations rather
than *registering* them). Both are pulled from the docs ../RESEARCH_NOTES.md
already cites, refreshed against live sources for this analysis.

---

## Ground truth the rest of this doc rests on

Before assessing each item, four facts the code shows that the docs
mis-state or under-state. These matter because they change which deferrals
are actually deferred and which are already shipped.

### 1. `BindingBuilder` IS the working API and the extensions DO use it

`../spec/BINDING_API.md`'s v0.3 annotation and `src/binding_builder.hpp`'s header
comment both describe `BindingBuilder` as the "interim floor" over a
`NativeSig` map, and frame `TypeBuilder`/`engine_t` as the fluent target the
extensions "move toward." The code shows the move is **already done** for the
registration floor and is not in flight:

Every one of the six extensions includes `binding_builder.hpp` and registers
via `BindingBuilder::add` / `add_overload`, not the old `I/H/add` lambda
boilerplate the header comment references. Concretely (`ext_vec.cpp`'s
`register_natives`):

```cpp
BindingBuilder b;
b.add("vec3_new", bind_handle("vec3"), {type_f32(),type_f32(),type_f32()}, (void*)&n_vec3_new);
b.add("vec3_x",   type_f32(), {type_i64()}, (void*)&n_vec3_x);
...
NativeTable t = b.build();
m = std::move(t.natives);
```

and `register_overloads`:

```cpp
b.add_overload("vec3", int(BinExpr::Op::Add), bind_handle("vec3"), (void*)&n_vec3_add);
```

`ext_array`, `ext_mat`, `ext_math`, `ext_quat`, `ext_string` are identical in
shape. The "six copies of I/H/add deduped" claim in the header is accurate as
history but the dedup **has landed**; the header comment describing the
current code as "the floor the extensions move toward" is stale — the
extensions are already on it. This is the single most important correction
for the rest of the analysis: `BindingBuilder` is not a transitional shim
awaiting `TypeBuilder`; it is the API the shipped addon set actually builds
on, and it covers the full surface the six extensions exercise.

### 2. The `PERM_FFI` compile-time gate is ALREADY implemented in sema

`../spec/BINDING_API.md`'s v0.3 annotation and `src/binding_builder.hpp`'s
permission comment both state: "v0.3 stores the bit on the `NativeSig`; full
permission gating in codegen is deferred to the v0.4 safety milestone — v0.3
does not yet check-and-refuse before marshalling." The code contradicts this.

`src/sema.cpp` line 393, inside the native-call resolution branch:

```cpp
if (nit != natives->end()) {
    // permission gate (../spec/SAFETY_AND_SANDBOX.md Section 6)
    if ((nit->second.permission & 0x01 /*PERM_FFI*/) && !(perms & 0x01)) {
        err("function '" + c->name + "' requires PERM_FFI", c->loc.line, c->loc.col);
    }
    c->is_native = true; c->native_fn = nit->second.fn_ptr;
    ...
```

This is exactly the gate `../spec/SAFETY_AND_SANDBOX.md` Section 6 specifies:
"compile time, at the call-site sema check ... a **compile error**, not a
runtime check ... zero-cost at runtime (no branch ever emitted for it)." It
fires before `c->native_fn` is stamped and before any arity/type/marshalling
check. There is no codegen-level gate and there should not be one — the spec
explicitly scopes the gate to compile time. The doc annotations ("v0.3 does
not yet check-and-refuse," "codegen-level gating is v0.4") are **wrong about
the shipped code**; the gate matches the v1.0 spec already.

Why nobody has noticed: every shipped exe (`ember_cli`, `sema_check`,
`ext_registration_test`, `em_roundtrip_test`, `ext_runtime_test`) passes
`module_permissions = 0` to `sema()`, and no shipped extension registers any
native with `PERM_FFI`. So the gate's branch is dead code in the current test
surface — the bit is never set on a `NativeSig` the gate would trip against.
The gate is shipped, correct, and unexercised. The deferral annotation is a
documentation artifact of an earlier state that was overtaken by the sema
change without the doc being updated.

Two minor notes on the gate's quality (not deferrals, just things to tighten
when the gate is first exercised):

- After `err()` the code **continues** to stamp `c->is_native = true` and
  `c->native_fn` and runs the arity/type check. A genuine permission
  violation thus produces the PERM_FFI error *plus* possibly a cascading
  arity error if the gated native also has a signature mismatch. Harmless
  in practice — every caller checks `sr.ok` and exits before codegen — but
  the gate should `return` early (or stamp a sentinel) after recording so
  the violation is one clean error, matching SAFETY Section 6's "different
  error codes/messages so host tooling can distinguish 'not allowed' from
  'doesn't exist'" intent.
- The gate reads only bit `0x01` against `perms & 0x01`. `PERM_FFI` is the
  only permission defined, so this is correct today; it is not a
  bitmask-generic gate, just a single-bit test. Fine for v1.0's one
  permission, but if a second permission bit is ever added this needs to
  become a proper `(permission & ~module_perms) != 0` check.

### 3. The overload model is complete for the i64-handle pattern the extensions use

`BindingBuilder::add_overload` hardcodes the overload's parameter shape as
`{bind_prim(Prim::I64), bind_prim(Prim::I64)}` — i.e. it assumes both
operands are i64 handles. This looks like a limitation until you read
`sema.cpp`'s overload dispatch (line ~578):

```cpp
if (overloads && lt && rt && lt->same(*rt)) {
    std::string tname = lt->struct_name;
    if (!tname.empty()) {
        const OpOverload* oo = overloads->find(tname, int(b->op));
        if (oo) { b->is_overload = true; b->overload_fn = oo->fn_ptr; ... }
    }
}
```

The dispatch fires only when both operands share a `struct_name`. For every
shipped extension that struct_name tags an **i64 handle** into a host-owned
store (`vec3` = i64 index into `g_vec3s`, `string` = i64 index into
`g_strings`, `mat4` = i64 index into `g_mat4s`, etc.). So an overload fn
genuinely does take `(i64 lhs_handle, i64 rhs_handle) -> ret`. The hardcoded
`(i64,i64)` param shape in `add_overload` is **correct for the only overload
shape the language can currently express**, not a workaround. A
`TypeBuilder.bin_add` that let you register a struct-by-value overload would
be dead API today: sema's dispatch path only matches i64-handle-typed
operands, and `check_struct_arg_shape` requires a struct-by-value argument to
be a bare local — there is no path where two struct-by-value locals flow into
a `BinExpr` that the overload table could intercept.

### 4. The CLI/test exes do ~20 lines of manual setup each, and it is almost all engine_t-shaped

Counting `register_natives` + `register_overloads` + `build_struct_layouts`
+ `sema` + globals-block wiring + dispatch-table + `CodeGenCtx` wiring +
`compile_func`/`finalize`/`table.set` per function, the per-exe setup cost:

| Exe | setup-wiring lines |
|---|---|
| `ember_cli.cpp` | 28 |
| `sema_check.cpp` | 19 |
| `ext_registration_test.cpp` | 16 |
| `em_roundtrip_test.cpp` | 18 |
| `ext_runtime_test.cpp` | 12 |

`ember_cli.cpp`'s 28 lines are the most instructive. They fall into two bands:

- **~10 lines that are genuinely per-host policy** (slot assignment, globals
  block sizing, dispatch table construction, finding the entry function,
  calling it): these are the host's *execution* policy and an `engine_t`
  that hides them would be hiding decisions a host legitimately wants to own
  (which function is the entry, how globals are backed, whether to dump).
- **~18 lines that are pure boilerplate ceremony** (`register_natives` x6,
  `register_overloads` x4, `build_struct_layouts`, `sema(prog, natives,
  slots, 0, &overloads, &struct_layouts)`, the `CodeGenCtx` field
  assignments `ctx.globals_base = ...; ctx.dispatch_base = ...;
  ctx.natives = &natives; ctx.script_slots = &slots; ctx.structs =
  &struct_layouts;`). This is the ceremony an `engine_t`-shaped wrapper
  would collapse: the host says "compile and run this program with the
  standard addon set" and the wrapper owns the maps, the layout table, the
  ctx wiring, the per-function compile/finalize loop, the dispatch table.

So the engine_t re-entry trigger question is concrete and answerable from
the code: the standard addon surface is registered identically in **three**
places (`ember_cli`, `sema_check`, `ext_registration_test`), with the same
six `register_natives` calls and the same four `register_overloads` calls in
each. That is real triplication, and it is the kind of triplication an
`engine_t` (or even just a `register_standard_addons(Engine&)` helper short
of a full engine_t) exists to kill.

## Per-item analysis

### TypeBuilder

**What it is.** A fluent per-type builder (`TypeBuilder(engine, name, size,
align).field(...).method(...).property(...).bin_add(fn)...finish()`) that
declares a host C++ struct as a script-visible type with fields, methods,
properties, and operator overloads all registered against one named type in
one RAII block. Per `../spec/BINDING_API.md` Section 3, methods and properties
desugar to ordinary native calls where the receiver is synthesized as the
first `NativeParam`; operators feed the same overload table the current
`add_overload` feeds. It is the surveyed native-JIT language's
`type_builder` surface trimmed to v1's reachable language features
(subscript/iterable/factory/hash/convert/destructor dropped as YAGNI per
feature, each with a documented re-entry trigger tied to a grammar addition).

**What it would take to build.** Concrete:

- New header `src/type_builder.hpp` (~250 LOC): the `TypeBuilder` class
  itself — a builder that accumulates `(name, offset, type)` field tuples,
  `(name, fn, ret, params)` method tuples, `(name, getter, setter, type)`
  property tuples, and a per-op overload map, then in `finish()` (a) writes
  a `StructLayout` into a `StructLayoutTable` with the field offsets and
  types, and (b) writes each method as a `NativeSig` into the natives map
  (synthesizing the receiver as `NativeSig.params[0]`) and each operator as
  an `OpOverload`. No new sema or codegen types: `StructLayout` already
  exists, `NativeSig` already exists, `OpOverload` already exists —
  `TypeBuilder` is sugar over the three maps `BindingBuilder` already fills,
  plus a `StructLayout` entry.
- The method-desugar needs one small sema change: today `CallExpr::receiver`
  is set by the **parser** for `obj.method(args)` syntax (the method-call
  sugar), and sema resolves it by treating the receiver as `args[0]` and
  checking against `NativeSig.params[0]`. `TypeBuilder.method` would
  register methods whose `params[0]` is the struct handle; the existing
  receiver path already handles this, so no sema change is needed for method
  dispatch. The change is purely that `TypeBuilder` registers the method's
  `NativeSig` with the right `params[0]` shape.
- The `field()` path needs codegen awareness: a `FieldExpr` against a
  `TypeBuilder`-registered type needs the field's offset, which
  `StructLayoutTable` already provides. The existing `FieldExpr` codegen
  path (lines ~1290+ of `codegen.cpp`, the `flt->base` Ident lookup +
  `ctx.structs->find(bt->struct_name)` + `fit->second.offset`) already
  handles this for script-declared structs; a `TypeBuilder`-registered type
  just appears as another `StructLayout` entry. No codegen change needed.
- Property (`obj.x` get/set desugaring to a getter/setter native call) is the
  one piece that needs real new work: today `FieldExpr` is purely a
  field-offset load/store, with no notion of a property redirect. This needs
  a sema-level rewrite of `FieldExpr` to a `CallExpr` when the resolved
  `StructLayout` entry for that field name is marked `is_property`. Maybe
  ~60 LOC in sema + a flag on `StructFieldLayout`.

Realistic total: ~300 LOC of new builder + ~30 LOC of sema tweak for
properties. No codegen change beyond what `StructLayoutTable` already drives.

**The real re-entry trigger.** The roadmap's generic trigger is "a real host
wants script-visible C++ struct types." Grounded in the actual extension
code, the trigger is sharper and **has not fired**: every one of the six
extensions represents its host type as an **opaque i64 handle** into a
host-owned `std::vector<T>` store, not as a script-visible struct. `vec3` in
script is `i64` with `struct_name="vec3"` as a tag for overload resolution;
the script never sees `v.x` as a field — it calls `vec3_x(v)`. There is no
extension today that wants `v.x` field-access syntax over a host-owned Vec3,
because the host store is an opaque handle by deliberate design (the
"host-owned memory" ownership category in `../spec/MEMORY_AND_GC.md` Section 1,
which lets the host lifetime-manage the storage without ember needing a GC).

For the trigger to fire, a host would need to want **value-type struct
semantics** in script — construct a struct literal `Color{r:255,g:255,b:255,a:255}`,
pass it by value to a native, read `.r` directly, and have the struct live in
a script frame slot (not behind a host handle). That is a real and common
game-scripting pattern (color/point/rect as value types), and `sema.cpp`'s
`check_struct_arg_shape` + `codegen.cpp`'s struct-by-value stash path (the
`copy_bytes` from a bare local's frame address) already support it for
**script-declared** structs. The gap `TypeBuilder` would close is: a *host*
C++ struct (with a fixed layout the host wants to keep in C++, e.g. to pass to
a rendering API by value) registered as a script-visible value type with
field access — so the script can do `Color{...}` and the host can take that
`Color` by value into a real C++ `struct Color{u8 r,g,b,a;}` without an i64
handle indirection.

Assessment: **distant.** No extension needs it; the i64-handle model is
working and is the documented ownership model. The trigger fires when a host
shows up wanting a real value-type struct (Color/Point/AABB) the script
constructs and the host passes to a C API by value — not before.

**Honest gap assessment.** There is **no gap the extensions are working
around**. `BindingBuilder` + `add_overload` + the i64-handle convention cover
exactly the surface the six extensions exercise. `TypeBuilder` would add
value-type-struct ergonomics for a use case no shipped addon has. The YAGNI
stance holds; the deferral is correct.

### StructBuilder

**What it is.** A data-only subset of `TypeBuilder` —
`StructBuilder(engine, name, size, align).field(...).finish()`, no methods or
operators. Per `../spec/BINDING_API.md` Section 5, kept as a separate class
rather than folded into `TypeBuilder` because "this is a plain data struct,
no behavior" is useful declared intent.

**What it would take to build.** ~80 LOC: a strict subset of `TypeBuilder`
that only writes the `StructLayout` entry, no `NativeSig` or `OpOverload`
writes. Trivially derivable from `TypeBuilder` by deleting the method/property/
operator methods.

**The real re-entry trigger.** Same as `TypeBuilder` minus the operator
question — a host wants to expose a plain-data host struct as a
script-visible value type with field access. The six extensions don't want
this (they use handles). **Distant, and strictly weaker than TypeBuilder** —
if the value-type-struct trigger fires, the host almost certainly wants
methods too (a `Color` with fields usually wants `lerp(c1,c2,t)`), so
`TypeBuilder` is the natural build, with `StructBuilder` as a convenience
subcase only if a host specifically wants to forbid method registration as a
declared constraint.

**Honest gap assessment.** No gap today. `StructBuilder` is the smaller,
less-likely-to-be-built-alone sibling of `TypeBuilder`; building it without
`TypeBuilder` would be odd.

### EnumBuilder — language trigger fired; host builder still unnecessary

**What it is.** A host-side builder for a script-visible enum type. Per
`../spec/BINDING_API.md` Section 5, **explicitly dropped from v1**: "no `enum`
keyword in the v1 grammar. Host-side named constants use `set_global`. Revisit
if script-side enum syntax is added later."

**What it would take to build.** A script-side `enum` grammar addition (a
new `EnumDecl` AST node, parser support, sema that resolves enum-ident
references to their i32 value) **plus** a host-side `EnumBuilder` that
registers the enum's name→value map. The host side is cheap (~50 LOC); the
grammar+sema+codegen side is the real cost (~200 LOC across parser/sema/
codegen, plus a type-check rule that enum-typed values are i32 under the hood
and that `switch` on an enum scrutinee works). This is a Tier 1 *language*
feature (`../ROADMAP.md` Tier 1 lists `enum`), not a binding feature; the
`EnumBuilder` is the binding tail of a language change.

**The real re-entry trigger.** `../ROADMAP.md` Tier 1's trigger: "a real
script wants more than ~5 related constants and the global-flat hurts
readability." **The language trigger fired and source enums shipped (including typed enums).** No shipped host binding requires
more than a handful of related i32 constants; the closest thing in the tree is
the assertion-native set (`assert_eq_i64` etc.) which is namespaced by name
prefix, not enum. Assessment: **distant**, and gated on a language change, not
a binding gap. The binding-side `EnumBuilder` is the cheap tail; it should not
be built until the grammar lands, or it is dead API surface (../spec/BINDING_API.md
Section 5's own YAGNI rationale).

**Honest gap assessment.** No gap. The deferral is correct and is paired with
the language-side deferral in ROADMAP Tier 1.

### engine_t

**What it is.** A host-managed wrapper object bundling the `natives` map +
`overloads` table + `struct layouts` + `dispatch table` + `globals block` +
`module permissions` into one handle the host passes to `ember_compile` /
`ember_call` instead of threading six loose references through every call.
Per `../RESEARCH_NOTES.md`, this is the surveyed native-JIT language's
`engine_t` / `module_t` / `context_t` split mirrored directly, and per
`../spec/SAFETY_AND_SANDBOX.md` Section 6 it is also the mechanism that makes
per-module permission isolation expressible ("register the sensitive natives
on a separate `engine_t` instance used only for the trusted module's
compilation").

**What it would take to build.** New `src/engine_t.hpp` + `.cpp` (~200 LOC
total). The shape is almost exactly `ember_cli.cpp`'s body lines 130–210
extracted into a struct:

```cpp
struct engine_t {
    NativeTable bindings;            // natives map + overloads (BindingBuilder::build output)
    StructLayoutTable layouts;       // build_struct_layouts(prog) output
    DispatchTable dispatch;          // per-module, sized to prog.funcs.size()
    GlobalsBlock globals;            // sized to prog.globals
    CodeGenCtx ctx;                  // natives/overloads/structs/dispatch_base/globals_base wired
    uint32_t module_permissions = 0;
    // ... compile(prog) -> fills dispatch + layouts + ctx; call(fn_name) -> calls entry ...
};
```

plus convenience `engine_t::register_standard_addons()` that does the six
`ext_*::register_natives` + four `register_overloads` calls (this is the
helper that kills the three-way triplication in the exes, and it does not
need `engine_t` to exist — a free function
`register_standard_addons(BindingBuilder&)` delivers 90% of the dedup value).

No sema change, no codegen change. `engine_t` is pure host-side aggregation
over the existing `sema`/`compile_func`/`finalize`/`table.set` calls; it
changes call shape, not behavior.

**The real re-entry trigger.** The roadmap's generic trigger is the v1.0
milestone ("stable native binding API"). Grounded in the actual code, the
trigger is **the three-way triplication of the standard addon registration
across `ember_cli.cpp`, `sema_check.cpp`, and `ext_registration_test.cpp`**,
all of which run the same six `register_natives` + four `register_overloads`
calls. This trigger **has already fired** — the triplication exists today, and
it is exactly the kind of boilerplate an `engine_t` (or even just a
`register_standard_addons` helper) exists to collapse.

The harder question is whether the full `engine_t` is worth it or just the
helper. The 28 setup lines in `ember_cli.cpp` break into ~10 lines of genuine
host execution policy (slot assignment, globals backing, entry-fn selection,
dump flag, exit-code rule) and ~18 lines of ceremony that `engine_t` would
own. A bare `register_standard_addons(BindingBuilder&)` helper collapses the
10 registration lines (six `register_natives` + four `register_overloads`) —
the biggest single chunk of triplication — without hiding the execution
policy a host wants to own. The remaining ~8 lines of ctx-wiring ceremony
(`ctx.globals_base = ...` etc.) is what a full `engine_t` would additionally
collapse, at the cost of hiding the globals/dispatch backing decisions.

Assessment: **the helper has fired; the full `engine_t` is close but
costed.** The full `engine_t` pays back only when there is a *second*
embedding host (the existing in-tree host is one; a second consumer would be the
trigger for "this is now a reusable wrapper, not a host-shaped one"). Until
then, a host that wants its own globals backing / dispatch sizing / entry
convention (which is what `ember_cli` does — it uses the entry's return value
as the process exit code, a host policy choice) would have to fight `engine_t`
to get its policy back. So the recommendation (below) splits: ship the helper
now (it is pure win, zero policy loss), defer the full `engine_t` to the v1.0
"example game-engine integration" milestone where the embedding shape is real
and a second host exists.

**Honest gap assessment.** There is a **real, already-fired gap** — the
standard-addon registration is triplicated. But it is a ~10-line-per-site gap,
and the right fix is the small helper, not necessarily the full `engine_t`.
The full `engine_t`'s value proposition (per-module permission isolation via
separate `engine_t` instances) is not exercised by any shipped host today,
because no shipped host registers `PERM_FFI` natives — the
permission-isolation use case is latent, waiting on a real gated-binding
consumer.

### NativeFn / NativeParam / TypeId

**What they are.** The descriptor-struct form of the binding API from
`../spec/BINDING_API.md` Sections 1–2: `TypeId` (a `uint16_t` enum of
`{t_void, t_bool, t_i8..t_i64, t_u8..t_u64, t_f32, t_f64, t_slice, t_struct}`),
`NativeParam{name, type, type_name, elem_type}` (one per parameter, with
`type_name` carrying the struct name for `t_struct` and `elem_type` carrying
the slice element `TypeId`), `NativeFn{name, fn_ptr, ret, ret_type_name,
params, param_count, permission}` (the full native descriptor, one per
registered function), and `register_native(engine, const NativeFn&)`.

The shipped replacement is `NativeSig{name, fn_ptr, ret, params, permission}`
in `src/sema.hpp`, where `params` is `std::vector<Type>` and `Type`
(`src/ast.hpp`) carries `{prim, struct_name, is_slice, array_len, elem}` in
one struct instead of `NativeParam`'s `{type, type_name, elem_type}` three
fields. `NativeSig` is sema's internal representation; `BindingBuilder::add`
is the host's way to fill one.

**What it would take to build.** Three things, in increasing cost:

1. **The `TypeId` enum + `NativeParam` + `NativeFn` structs as the
   host-facing ABI** (~120 LOC in a new `src/binding_abi.h`): a C-friendly
   `#pragma once` header with the plain-C struct shapes from ../spec/BINDING_API.md
   Sections 1–2. This is the "non-C++ host / dynamically-generated bindings"
   surface the spec calls out — a C ABI a non-C++ language binding (a
   Rust/C#/Python FFI) could link against without pulling in `ast.hpp`'s
   `std::shared_ptr<Type>`. Today a non-C++ host has no stable ABI to link; it
   would have to construct `Type` objects, which are C++ types.
2. **A `register_native(engine_t*, const NativeFn&)` entry point** that
   converts a `NativeFn` into the internal `NativeSig` (~30 LOC).
3. **The Section 7 host-side template wrapper**
   `register_native_fn<&fn>(engine, "name")` (~100 LOC of `if constexpr`
   trait dispatch deriving `TypeId` per C++ parameter type) — the "descriptor
   can never drift from the pointer" mitigation.

(1)+(2) are the real deliverable; (3) is the additive nicety the spec already
marks as "v1.0-milestone-adjacent, not required."

**The real re-entry trigger.** The spec's framing is "the API must also
support raw registration for non-C++ hosts / dynamically-generated bindings."
Grounded in the actual code, the trigger is: **a non-C++ host wants to bind
natives to ember.** Today there is exactly one host language in the tree (C++,
across the host and the five example exes), and `BindingBuilder::add` is ergonomic
for that host. The descriptor-struct ABI's value is solely for a non-C++
consumer. **Has not fired** — no non-C++ host exists.

A secondary trigger is dynamically-generated bindings (a host that generates
native signatures at runtime from data, not from C++ function pointers) — also
not present in the tree.

Assessment: **distant, and the cost-benefit is worse than it looks.** The
descriptor ABI is "boring C structs for non-C++ hosts" — but ember's execution
model is Win64-only, Windows-first, x86-64 JIT. A non-C++ host on Windows
already needs a C++ toolchain to *build* ember (the JIT emitter, the
sema/codegen C++); the binding ABI being C++-ergonomic (`BindingBuilder::add`)
rather than C-descriptor is not a real barrier to a Rust/C# host, which can FFI
into `BindingBuilder` via a tiny C++ shim the same way it would FFI into
`register_native`. The descriptor structs are a real cost (~150 LOC) for a
hypothetical consumer that does not exist and would need a C++ shim anyway for
the build.

**Honest gap assessment.** No gap the shipped code is working around.
`NativeSig` (internal) + `BindingBuilder` (host-facing) cover the one host
language that exists. The descriptor-struct ABI is the right shape *if* a
non-C++ host appears, and the spec is correct to design it, but building it now
is YAGNI — it is API surface with no consumer. The one piece of this item that
is genuinely worth building is the **`TypeId`-as-`uint16_t` enum**, because a
future `engine_t` ABI exposed to non-C++ hosts wants a stable type-id
enumeration that is not `Prim` (an `enum class Prim` in `ast.hpp` that could
evolve). But that is a v1.0+ concern, not a v0.3 gap.

### Full permission gating in codegen (PERM_FFI enforcement)

**What it is.** The deferral as stated: "v0.3 stores the bit on the `NativeSig`;
full permission gating in codegen is deferred to the v0.4 safety milestone —
v0.3 does not yet check-and-refuse before marshalling."

**What it would take to build.** **Nothing — it is already built.** See
Ground Truth §2 above: `src/sema.cpp` line 393 implements the compile-time gate
exactly as `../spec/SAFETY_AND_SANDBOX.md` Section 6 specifies. There is no
codegen-level gate and the spec explicitly says there should not be one (the
gate is compile-time-only to be "zero-cost at runtime"). The v0.4 milestone
item "permission gating in codegen" is, on inspection of the code, a no-op: the
gate exists, is correct, and matches the v1.0 spec.

What v0.4 *should* own is not building the gate but **exercising and hardening
it**:

- The "continue after `err()`" cascade noted in Ground Truth §2.
- A test that registers a `PERM_FFI` native, compiles a module with
  `module_permissions = 0`, and asserts the sema error fires and the program
  does not reach codegen. No such test exists today (which is *why* the
  doc/code divergence went unnoticed).
- A test that compiles the same source with `module_permissions = PERM_FFI`
  and asserts the call resolves and runs.
- Decision: should `set_permissions` be an engine-wide knob (the spec's
  current model, matching the surveyed language's `set_permissions(engine,
  flags)`) or per-module (the `engine_t`-instance-per-module route SAFETY
  Section 6 mentions)? This is a real design question for v0.4, not a
  gate-implementation question.

**The real re-entry trigger.** The v0.4 safety milestone. **Already fired
for the gate's existence**; not yet fired for the gate's *exercise*. The
trigger for the hardening work is "v0.4 budgets/safety milestone starts," which
is the next milestone after v0.3. So: **imminent**.

**Honest gap assessment.** The gap as documented (gate not built) does not
exist. The real gap is that the gate is **untested and undocumented in the
shipped code** — the doc annotations actively claim it is not there, which is
worse than merely untested because it means a future maintainer reading
`binding_builder.hpp` would believe the gate is missing and might re-implement
it in codegen (where the spec says it must not go). The fix is documentation:
correct the stale annotations in `binding_builder.hpp` and `../spec/BINDING_API.md` to
say "compile-time gate shipped in v0.3 sema (`src/sema.cpp` line 393), matching
../spec/SAFETY_AND_SANDBOX.md Section 6; v0.4 owns the exercise/hardening test suite, not
the gate itself."

## Recommended v1.0 binding surface — honest, evidence-based

Ordered by what the code actually justifies building, with the spec
challenged where the evidence warrants.

### Build now (v0.4 / pre-v1.0): the `register_standard_addons` helper

This is the one piece the code unambiguously calls for. The standard addon
registration (six `register_natives` + four `register_overloads`) is
triplicated across `ember_cli.cpp`, `sema_check.cpp`, and
`ext_registration_test.cpp`, and the triplication is the real, already-fired
re-entry trigger for the `engine_t` item. A free function

```cpp
namespace ember { void register_standard_addons(BindingBuilder& b); }
```

in a new `src/standard_addons.hpp` (~30 LOC, links the six extension libs)
collapses the triplication without hiding any host execution policy. Every
exe becomes `BindingBuilder b; register_standard_addons(b); NativeTable t =
b.build();`. This is not the full `engine_t` — it is the 90%-of-the-value
slice, and it loses nothing a host wants to own. **Estimated cost: ~30 LOC.
No sema/codegen change. Build for v0.4.**

### Build for v1.0 (the milestone owns it): the full `engine_t`, gated on a second embedding host

The full `engine_t` is the v1.0 "stable native binding API" deliverable per
`DESIGN.md`'s milestone list. The evidence says: build it for v1.0, but
**do not build it before a second embedding host exists**, and frame it as
"the embedding wrapper the existing host + a second consumer both want," not "the v0.3
binding-API correctness wrapper." Today there is one consumer (the host, plus the
standalone CLI which is itself a consumer of the same shape); a
single-consumer `engine_t` would bake the host's execution policy (entry-fn
selection, globals backing, exit-code convention) into a wrapper that the
*next* consumer would then have to fight. The v1.0 "example game-engine
integration" milestone is exactly where the second consumer appears, so the
trigger and the milestone coincide. **Estimated cost: ~200 LOC. No
sema/codegen change. Build at v1.0, not before.**

### Build for v1.0 alongside `engine_t`: `TypeBuilder` and `StructBuilder`, IF a value-type-struct consumer appears

`TypeBuilder` is the most likely of the "fluent builder" trio to earn its
keep at v1.0, and only if a host wants script-visible value-type structs
(Color/Point/AABB constructed by literal, passed by value to a C API, field-
accessed in script). The trigger has not fired today; the six extensions are
all i64-handle-based and that model is the documented ownership model. The
v1.0 "example game-engine integration" milestone is the natural place to check
whether the example host wants value-type structs — a real game-engine binding
will likely want `Color{r,g,b,a}` as a value type, which is exactly the
`TypeBuilder` use case. **Recommendation: defer the build to v1.0, but
re-evaluate at the start of the v1.0 milestone against the example host's
actual struct needs. Build `StructBuilder` only if a host explicitly wants the
data-only constraint; otherwise `TypeBuilder` covers it.** Estimated cost:
~300 LOC + ~30 LOC sema tweak for properties.

### Keep deferring: `EnumBuilder`

It is gated on a Tier 1 language-side `enum` grammar change, not a binding
gap. The binding-side `EnumBuilder` is the cheap tail of a language change;
building it before the grammar lands is dead API surface (the spec's own YAGNI
rationale in ../spec/BINDING_API.md Section 5). **Keep deferred until ROADMAP Tier 1's
`enum` trigger fires.**

### Keep deferring, but split the item: `NativeFn`/`NativeParam`/`TypeId`

The spec is right that a stable C-ABI descriptor form is the correct shape *for
a non-C++ host*. The evidence says no such host exists and the cost (~150 LOC
of ABI surface + a template wrapper) is paid for a hypothetical consumer.
**Keep the full descriptor ABI deferred** until a non-C++ host appears.

The one sub-piece worth promoting is the **`TypeId` enum as a stable
`uint16_t`** — not as a host-facing ABI today, but as the type-id a future
`engine_t` ABI would expose. Today `Prim` (the internal primitive enum) serves
this role and is fine; the spec's `TypeId` adds `t_slice`/`t_struct` as
first-class kinds that `Prim` doesn't (a `Type`'s slice/struct-ness is
expressed via `is_slice`/`struct_name` fields, not the enum). If `engine_t`
ships at v1.0, its introspection surface (a `type_id_of(Type)` queryable by a
host) wants `TypeId`, not `Prim`. **Recommendation: build `TypeId` as part of
the v1.0 `engine_t` work if `engine_t` includes an introspection surface;
otherwise keep deferring with the rest of the descriptor ABI.**

### Already shipped — fix the docs: `PERM_FFI` compile-time gating

The gate is built (`src/sema.cpp` line 393), matches the v1.0 spec
(../spec/SAFETY_AND_SANDBOX.md Section 6), and is compile-time-only as the spec
requires (zero-cost at runtime, no codegen gate, which is correct). The
deferral annotation is a stale documentation artifact. **Fix the docs** in
two places:

- `src/binding_builder.hpp` lines 20–22 and 32–36: change "full permission
  gating in codegen is v0.4 safety; v0.3 just defines the constant" to "the
  compile-time gate is shipped in v0.3 sema (`src/sema.cpp` line 393),
  matching ../spec/SAFETY_AND_SANDBOX.md Section 6; v0.4 owns the exercise/hardening
  test suite, not the gate itself."
- `../spec/BINDING_API.md`'s v0.3 annotation's `PERM_FFI` paragraph: same
  correction.

Then v0.4's real scope for this item is: write the two tests (gated native
called from a non-permitted module → sema error, no codegen; same native from a
permitted module → runs), fix the "continue after `err()`" cascade so the
violation is one clean error, and decide the engine-wide-vs-per-`engine_t`
permission-knob question (which is really the `engine_t` design question, see
above).

## Prior-art grounding

### AngelScript — what "bindings like AngelScript" actually means

AngelScript's host-facing registration surface (verified against the live
SDK docs at `angelcode.com/angelscript/sdk/docs/manual/doc_register_func.html`
and `doc_register_type.html`):

- `RegisterGlobalFunction(decl_string, fnptr, callConv, aux)` — registers a
  free function, parsed from a declaration string like
  `"void print(i64&in)"`.
- `RegisterObjectType(decl_string, byteSize, flags)` — registers a type
  (reference or value), from a decl string like `"Color"`.
- `RegisterObjectMethod(objType, decl_string, fnptr, callConv, aux)` —
  registers a method on a registered type, decl-string parsed.
- `RegisterObjectProperty(objType, decl_string, byteOffset)` — registers a
  field, decl-string + offset.
- `RegisterObjectBehaviour` — registers operator overloads and other
  behaviours, decl-string parsed.

The defining property of AngelScript's API is that **every registration takes
a declaration string that a mini-parser parses at registration time**, and
that the call dispatch uses **per-platform hand-written native trampolines**
(`as_callfunc_x64_msvc.cpp` / `as_callfunc_x64_gcc.cpp`) to bridge the bytecode
VM's calling convention to arbitrary C++ function signatures. The
../RESEARCH_NOTES.md verdict — "decl-string parsing is the single biggest source
of AngelScript registration-time complexity ... and we don't need it" — is
correct: ember deliberately rejects decl-string parsing in favor of descriptor
structs (`NativeSig` / `Type`), and emits native calls as direct JIT'd
`mov`+`call` (no trampoline layer, since ember is already emitting real
x86-64).

So "bindings like AngelScript" means the *ergonomic shape* — one declarative
call per native, one declarative call per type with its fields/methods/
operators — **not** the decl-string mechanism. `BindingBuilder::add(name, ret,
{params}, &fn)` is the ergonomic-shape equivalent of
`RegisterGlobalFunction("sig", &fn)`; `add_overload(type, op, ret, &fn)` is
the equivalent of `RegisterObjectBehaviour`. What `BindingBuilder` does *not*
have today that AngelScript does: the per-type fluent registration of
fields/methods/properties against one named type in one RAII block — that is
exactly `TypeBuilder`, the deferred item. So "like AngelScript" is *partially*
satisfied today (the free-function and operator surface), and the per-type
fluent surface is the part the deferral covers. The evidence says the per-type
fluent surface is the part no shipped addon needs (i64 handles cover the
type-tagged-operator case), so "like AngelScript" is satisfied *for the use
case the six extensions actually have*, and TypeBuilder is the satisfaction
for the use case no shipped extension has yet.

### Mun — the AOT-ABI alternative the spec cites

Mun's binding model (verified against the live `mun_abi` crate source at
`github.com/mun-lang/mun/main/crates/mun_abi/src/`): the host does **not**
register natives at all. Mun source declares `extern fn`s; the AOT compiler
emits a `.munlib` shared object carrying a `#[repr(C)]` ABI of
`FunctionDefinition{name: *const c_char, signature: FunctionSignature{
arg_types: *const TypeId, return_type: TypeId, num_arg_types: u16},
fn_ptr}` and a `StructMemory{fields, ..}` layout struct. The host *discovers*
these from the loaded module and calls through `fn_ptr`. There is no host-side
`TypeBuilder` at all — the type layout is authored in Mun source and the ABI
carries it.

ember's model is the inverse: the host *registers* natives (because ember
JIT-compiles script in-process, not AOT-loads a `.dll`), so the host owns the
binding declarations. This is why ember's deferred surface is `TypeBuilder`/
`NativeFn`-shaped (host authors the type/binding) rather than Mun-shaped (host
discovers the type/binding from a compiled ABI). The `engine_t` deferral
specifically mirrors Mun's `engine`/`module`/`context` *aggregation shape*
(../RESEARCH_NOTES.md: "engine/module/context split — same shape as AngelScript"
and Mun), not Mun's AOT-discovery model.

The honest read: ember is closer to AngelScript's host-registration model
than to Mun's host-discovery model, and its deferred `TypeBuilder` is the
AngelScript-shaped fluent-per-type API that the i64-handle convention makes
unnecessary for the six shipped addons. The `engine_t` deferral is the
Mun/AngelScript-shaped engine/module/context aggregation that pays back only
with a second consumer — which is the v1.0 milestone's job to produce.
